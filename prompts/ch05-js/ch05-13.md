> **Prompt**: When should I use event delegation instead of adding individual event listeners?

**Event Delegation** is a technique where you attach a single event listener to a parent element to manage events for all of its current (and future) children. This works because of **Event Bubbling**, where an event on a child "bubbles up" the DOM tree to its ancestors.

You should use event delegation in three specific scenarios:

---

## 1. When Handling Dynamic Content
If you are building an app where items are added or removed frequently (like a Todo list or an infinite scroll feed), individual listeners are a nightmare.
* **Without Delegation:** Every time you add a new `<li>`, you must manually run `addEventListener` on it. If you forget, the new item won't work.
* **With Delegation:** You attach one listener to the `<ul>`. It doesn't matter if you add 1 or 1,000 items later; the parent is already "listening" for clicks bubbling up from any child.

---

## 2. To Improve Memory & Performance
Every event listener you create consumes a small amount of memory. 
* **The Problem:** If you have a table with 5,000 rows and you attach a click listener to every single `<td>`, you are creating 5,000 listener objects. This can slow down the page and increase the memory footprint.
* **The Solution:** Attach **one** listener to the `<table>`. This uses significantly less memory and keeps the browser's "to-do list" (the Event Registry) clean.



---

## 3. Simplifying Code Logic
Delegation centralizes your logic. Instead of having listeners scattered across dozens of elements, you have one "manager" function.

### How to Implement it
Since the event bubbles up, you need to check which specific element was actually clicked using `event.target`.

```javascript
const list = document.querySelector('#main-menu');

list.addEventListener('click', (event) => {
  // Use .closest() to ensure we caught the right element
  const item = event.target.closest('li');

  if (item) {
    console.log('You clicked item:', item.textContent);
    // Perform logic here
  }
});
```

---

## Comparison Table

| Feature | Individual Listeners | Event Delegation |
| :--- | :--- | :--- |
| **Setup** | Easy for 1 or 2 static elements. | Requires a "target check" (if statement). |
| **Memory** | High (one per element). | **Low (one per parent).** |
| **Dynamic Items** | Must be added manually to new items. | **Works automatically for new items.** |
| **Complexity** | Becomes messy with many elements. | Centralized and cleaner for lists/grids. |

---

## When NOT to use it
* **Focus/Blur Events:** These events do not bubble by default (though you can use `focusin` and `focusout` as alternatives).
* **Complex Interactions:** If children have wildly different behaviors that don't share any logic, a single "manager" function might become an unreadable mess of `if/else` statements.
