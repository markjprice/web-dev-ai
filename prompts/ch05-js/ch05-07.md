> **Prompt**: How do I debug JavaScript using Chrome DevTools step by step? Show me how to use breakpoints in JavaScript and why they are useful.

Debugging with `console.log()` is fine for quick checks, but **Breakpoints** are the professional way to debug. They allow you to "freeze time" and inspect the exact state of your application at any given moment.

---

## 1. Why Use Breakpoints Instead of Log?
* **Live Inspection:** You can see every variable’s value in the current scope without typing individual log statements.
* **The Call Stack:** You can see exactly which functions were called to get to the current line.
* **Step-by-Step Execution:** You can run your code one line at a time to see exactly where logic fails.
* **State Manipulation:** You can actually change variable values while the code is paused to test fixes in real-time.

---

## 2. Step-by-Step: Setting a Breakpoint

### Step A: Open the Sources Tab
1.  Open your website in Chrome.
2.  Right-click anywhere and select **Inspect** (or press `F12`).
3.  Click the **Sources** tab at the top.

### Step B: Locate Your Script
1.  In the left-hand Navigator pane, find the `.js` file you want to debug.
2.  Click it to open the code in the center editor.

### Step C: Toggle the Breakpoint
1.  Find the line of code where you suspect a bug.
2.  Click the **line number** (the gutter) on the left. 
3.  A blue/arrow icon will appear. This means the browser will now pause whenever it hits this line.



---

## 3. Navigating Paused Code
Once the code hits your breakpoint, the browser pauses, and the line will be highlighted in yellow. You now have several "Step" controls in the top-right:

* **Resume (Play icon):** Continues running the script until it hits another breakpoint.
* **Step Over (Curved arrow):** Executes the next line of code but doesn't "enter" any functions called on that line.
* **Step Into (Down arrow):** If the line calls a function, this takes you *inside* that function's code.
* **Step Out (Up arrow):** Finishes the current function and returns to the caller.

---

## 4. Useful Debugging Panels
While paused, keep an eye on these sections in the right-hand sidebar:

### **Watch**
Click the **+** icon to add a specific variable name. DevTools will track its value throughout the entire debugging session.

### **Scope**
This shows you every variable currently available. It is divided into **Local** (inside the function), **Script** (let/const outside), and **Global** (window).

### **Call Stack**
This is a "history" of function calls. If your function `calculateTotal()` crashed, the Call Stack shows you that it was called by `submitOrder()`, which was called by `onClick()`. Clicking these names jumps you to those lines.



---

## 5. Pro Tip: The `debugger` Keyword
If you don't want to hunt through the Sources tab, you can hardcode a breakpoint directly into your JavaScript file:

```javascript
function complexMath(a, b) {
  const result = a * b;
  debugger; // The browser will automatically pause here if DevTools is open
  return result;
}
```

This is incredibly useful for debugging triggers that happen very quickly or on page load.
