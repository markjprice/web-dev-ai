> **Prompt**: How does the browser render the DOM and when does JavaScript run during page load? What happens if JavaScript tries to access DOM elements before they exist?

The process of turning a string of HTML into a functional webpage is a highly choreographed sequence known as the **Critical Rendering Path**. Understanding how the browser parses code and where JavaScript (JS) fits in is essential for avoiding common bugs.

---

## 1. How the Browser Renders the DOM
The browser doesn't wait for the entire HTML file to download before it starts working. It processes the document in stages:

1.  **Conversion & Tokenization:** The browser reads raw bytes and converts them into characters based on the file's encoding. It then identifies "tokens" (like `<html>`, `<body>`, etc.).
2.  **DOM Tree Construction:** As the browser finds start tags, it creates objects (Nodes) and links them into a tree structure called the **Document Object Model (DOM)**.
3.  **CSSOM Construction:** Simultaneously, the browser processes CSS to build the **CSS Object Model**.
4.  **The Render Tree:** The browser combines the DOM and CSSOM. This tree only contains visible elements (it ignores `<head>` and elements with `display: none`).
5.  **Layout (Reflow):** The browser calculates the exact geometry—where every pixel of every element should go on the screen.
6.  **Painting:** Finally, the browser fills in pixels (colors, borders, images) to display the content.



---

## 2. When Does JavaScript Run?
By default, JavaScript is **parser-blocking**. When the HTML parser encounters a `<script>` tag:
* It **pauses** the construction of the DOM.
* If the script is external, it must fetch it from the network.
* The JS engine executes the code immediately.
* Only after the script finishes does the parser continue building the DOM.

### Modern Optimization
You can change this behavior with attributes:
* **`async`**: The script downloads in the background and executes the moment it’s ready, regardless of where the parser is.
* **`defer`**: The script downloads in the background and only executes **after** the DOM is fully parsed, but before the `DOMContentLoaded` event.

---

## 3. Accessing Elements Before They Exist
If JavaScript tries to access a DOM element that the parser hasn't reached yet, it results in a **runtime error**.

### The "Null" Problem
If your script is in the `<head>` and tries to find a button in the `<body>`:
1.  The browser calls `document.getElementById('myButton')`.
2.  Since the parser hasn't reached the `<body>` yet, that element doesn't exist in the DOM tree.
3.  The method returns `null`.
4.  If you try to call a property on it (like `.addEventListener`), the browser throws a **TypeError**: *"Cannot read properties of null."*

### How to Prevent This
1.  **Move Scripts to the Bottom:** Place your `<script>` tags just before the closing `</body>` tag. This ensures the entire DOM is ready.
2.  **Use `defer`:** This tells the browser not to execute the script until the DOM is finished.
3.  **Event Listeners:** Wrap your code in an event listener that waits for the DOM to be ready:
    ```javascript
    document.addEventListener('DOMContentLoaded', () => {
        // Your code here is safe to run
    });
    ```
