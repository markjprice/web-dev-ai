> **Prompt**: Please explain the difference between `textContent`, `innerText`, and `innerHTML` with examples and security implications.

These three properties are often used interchangeably, but they behave differently under the hood—especially regarding performance, styling, and security.

---

## 1. The Definitions

### **`textContent`**
Returns the raw text content of all elements, including `<script>` and `<style>` elements. It is essentially a "dumb" dump of all text within the node.

### **`innerText`**
Returns the "rendered" text content. It is aware of CSS styling and won't return text that is hidden (e.g., `display: none`). It is essentially what a user would see if they highlighted the text on the page.

### **`innerHTML`**
Returns the full HTML markup contained within the element. It allows you to retrieve or insert tags like `<div>`, `<span>`, or `<strong>`.

---

## 2. Practical Example
Imagine this HTML structure:
```html
<div id="demo">
  Hello <span style="display: none;">Secret</span> 
  <strong>World!</strong>
</div>
```

If we access the `#demo` element, here is what each property returns:

| Property | Result | Why? |
| :--- | :--- | :--- |
| **`textContent`** | `"Hello Secret World!"` | It ignores CSS; it sees all text nodes. |
| **`innerText`** | `"Hello World!"` | It respects CSS; it hides "Secret". |
| **`innerHTML`** | `"Hello <span style="...">Secret</span> <strong>World!</strong>"` | It returns the actual tags and attributes. |



---

## 3. Performance & Layout
* **`textContent`** is the fastest. It simply gets the text from the DOM.
* **`innerText`** is slower. Because it needs to know what is "visible," it triggers a **reflow** (the browser has to calculate the layout and styles) to ensure it's giving you the correct rendered text.
* **`innerHTML`** is the heaviest if you are setting content, as the browser has to parse the string into new DOM nodes.

---

## 4. Security Implications (XSS)
This is the most critical difference. Using `innerHTML` with user-provided data is a major security risk called **Cross-Site Scripting (XSS)**.

### **The Risk with `innerHTML`**
If you take a username from a form and inject it using `innerHTML`, an attacker could provide a malicious script:
```javascript
// DANGEROUS: If user provides a script
const userInput = "<img src='x' onerror='alert(\"Hacked!\")'>";
element.innerHTML = userInput; // The browser executes the script!
```

### **The Safety of `textContent`**
`textContent` and `innerText` do not parse strings as HTML. They treat everything as literal text. If you use `textContent`, the tags are displayed as plain text on the screen rather than being executed by the browser.

```javascript
// SAFE: The browser displays the literal string, it doesn't run it.
element.textContent = userInput; 
```



---

## Summary Recommendation
* Use **`textContent`** by default for updating text (it’s fast and safe).
* Use **`innerText`** only if you specifically need to respect CSS visibility.
* Use **`innerHTML`** only when you intentionally need to insert HTML tags, and **never** with raw user input (always sanitize it first).
