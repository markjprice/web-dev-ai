> **Prompt**: Why does running the same Bash command sometimes work in one directory but fail in another?

When a command works in one place but fails in another, it’s usually because of the **environment state** or how the command interacts with the **filesystem**.

Since Bash is "context-aware," its behavior is tied directly to your current location (the **Working Directory**). Here are the three most common reasons for this discrepancy:

---

## 1. Relative vs. Absolute Path Issues

This is the number one cause of "it worked a minute ago" errors. If your command refers to a file using a **relative path**, that path is only valid if you are in the correct starting position.

* **Scenario:** You have a script in `/home/user/project/run.sh` that tries to read `config.json`.
* **Success:** You are inside `/home/user/project`. You run `./run.sh`. It finds the file.
* **Failure:** You `cd ..` into `/home/user`. You run `./project/run.sh`. The script looks for `config.json` in `/home/user` (where you are now), doesn't find it, and crashes.

---

## 2. Permissions and "Boundary" Issues

Your user might have permission to execute a command in your `Documents` folder, but not in a system folder like `/etc` or `/var`.

* **Ownership:** You might own the files in your Home directory, allowing you to run `rm *`. In a shared directory, those same files might be owned by `root`.
* **The Execute Bit:** As we discussed with `chmod +x`, a script must have the execute bit set. If you copy a script to a USB drive formatted with a filesystem that doesn't support Unix permissions (like FAT32), the execute bit might "disappear," causing the command to fail.

---

## 3. The "Hidden" Dot-Command Problem

Sometimes, the command itself is actually a file sitting in your current directory.

* **The `PATH` behavior:** By default, Linux does **not** look in your current folder for commands for security reasons.
* **Success:** You type `./my_script`. The `./` explicitly tells Bash: "Look right here in this folder."
* **Failure:** You type `my_script` without the `./`. Bash searches through your `$PATH` (like `/bin` and `/usr/bin`), doesn't find it, and says `command not found`—even though you are staring right at the file in your terminal!

---

## 4. Environment-Specific Configs

Some commands look for local configuration files (like `.env`, `package.json`, or `.git`) to determine how to behave.

* **Example:** Running `git status`.
* **Success:** Works inside any folder that is part of a Git repository.
* **Failure:** Fails with "fatal: not a git repository" if you move one folder higher than the project root.

---

### How to Debug This

If a command is failing, run these three "detective" commands to see what changed:

1. **`pwd`**: "Print Working Directory" – confirm exactly where you are.
2. **`ls -la`**: Check if the files you expect to be there actually exist and have the right permissions.
3. **`which <command>`**: Verify if the command is a system-wide program or a local script.
