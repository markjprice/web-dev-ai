> **Prompt:** Show a realistic Git branching strategy for a small Django team.

For a small Django team in 2026, the industry has shifted away from the heavy ceremony of "Gitflow" toward a **Hybrid Trunk-Based Development** model (often called "GitHub Flow with Environments").

This strategy prioritizes speed and automated testing while maintaining a high-quality production bar.

---

### The 2026 "Small Team" Branching Model

This model uses one permanent branch (`main`) and short-lived "Task" branches.

#### 1. The `main` Branch (The Source of Truth)

* **Rule:** Always deployable.
* **Logic:** This branch represents the state of your **Staging** or **Production** environment.
* **Protection:** In 2026, you should enable **Branch Protection Rules** on GitHub/GitLab to require:
* At least one approving review.
* Passing CI/CD status checks (Linting, Unit Tests, Security Scans).
* Linear history (Squash and Merge).



#### 2. Short-Lived Task Branches (`feat/`, `fix/`, `chore/`)

* **Rule:** Live for hours or 1–2 days max.
* **Naming Convention:** `feat/issue-number-description` (e.g., `feat/102-add-stripe-webhook`).
* **Workflow:** 1. Create branch from `main`.
2. Commit small, logical chunks.
3. Push to remote frequently to trigger CI feedback.

---

### The Step-by-Step Workflow

| Step | Action | Why? |
| --- | --- | --- |
| **1. Sync** | `git checkout main && git pull` | Start from the most recent stable code. |
| **2. Branch** | `git checkout -b feat/add-logging` | Isolate your work. |
| **3. Code** | Write code + tests. | Ensure your "Verification" logic is included. |
| **4. PR** | Open a **Draft Pull Request**. | Signals to the team you are working on this; triggers CI. |
| **5. Review** | Peer review + CI passing. | The human and machine "gates" for quality. |
| **6. Merge** | **Squash and Merge** to `main`. | Keeps your `main` history clean and easy to revert. |
| **7. Deploy** | Automated CD to Production. | Reduces the "lead time" between code and value. |

---

### Special Handling for Django Migrations

The biggest risk for Django teams is **Migration Conflicts** (two devs creating `0002_...` simultaneously).

* **The 2026 Fix:** Use a **CI check** that fails if there are multiple "leaf" migrations or if `makemigrations --check` detects missing files.
* **Strategy:** If you encounter a conflict, the dev merging *last* is responsible for rebasing, deleting their migration, and re-running `makemigrations` to ensure a linear sequence.

---

### Why this works for Small Teams

1. **Low Cognitive Overhead:** No "Develop" vs "Main" confusion.
2. **No "Integration Hell":** Because branches are short-lived, you rarely deal with massive 100-file merge conflicts.
3. **Automation First:** It forces the team to rely on a solid test suite rather than manual "QA branches."

### When to add a `release` branch?

Only move away from this if you have **scheduled weekly releases** or **regulatory requirements** (e.g., Healthcare/Finance) that require a manual sign-off period on a frozen set of code. Otherwise, stay on Trunk-Based development for maximum velocity.
