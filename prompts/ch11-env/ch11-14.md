> **Prompt:** Please show me a wait-for-database script for Django containers.

In a Docker environment, the database container often "starts" before the database engine inside it is actually ready to accept connections. If your Django container tries to run migrations immediately, it will crash with a connection error.

To solve this, we use a **"Wait-for-Database"** script. While there are bash-based versions using `netcat` or `pg_isready`, a **Python-based script** is often better because it uses the same environment and credentials as your Django app.

---

## 1. The Python "Wait-for-DB" Script

Create a file named `wait_for_db.py` in your project root (next to `manage.py`):

```python
import time
import sys
from django.db import connections
from django.db.utils import OperationalError
from django.core.management import execute_from_command_line

def wait_for_db():
    print("Waiting for database...")
    db_conn = None
    while not db_conn:
        try:
            # Try to get the default database connection
            db_conn = connections['default']
            # Force a connection check
            db_conn.cursor()
        except OperationalError:
            print("Database unavailable, waiting 1 second...")
            time.sleep(1)

    print("Database available!")

if __name__ == "__main__":
    # Initialize Django settings before checking the DB
    import os
    import django
    os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'your_project_name.settings')
    django.setup()
    wait_for_db()

```

## 2. Using it in Docker Compose

You should integrate this script into your container's startup command. The best way is to use a shell wrapper or an `entrypoint.sh` script.

### Option A: The Entrypoint Script (`entrypoint.sh`)

```bash
#!/bin/sh

# Exit immediately if a command exits with a non-zero status
set -e

echo "Checking database readiness..."
python wait_for_db.py

echo "Running migrations..."
python manage.py migrate

echo "Starting server..."
exec "$@"

```

### Option B: Inline in `docker-compose.yml`

If you don't want a separate entrypoint file, you can chain the commands in your YAML:

```yaml
services:
  web:
    build: .
    command: >
      sh -c "python wait_for_db.py &&
             python manage.py migrate &&
             python manage.py runserver 0.0.0.0:8000"
    depends_on:
      - db
  db:
    image: postgres:15

```

---

## 3. Why `depends_on` isn't enough

A common misconception is that `depends_on` handles this.

* **`depends_on`** only ensures the **container** is running.
* It does **not** know if the **Postgres/MySQL process** inside that container has finished initializing its data files and is listening on the port.

By using the script above, you ensure Django won't even attempt a migration until the database "handshake" is confirmed.

---

## 4. Pro-Tip: Using `healthcheck`

In modern Docker (2026), you can also use the native `healthcheck` feature in your database service and make your web service wait specifically for the **healthy** state.

```yaml
services:
  db:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  web:
    build: .
    depends_on:
      db:
        condition: service_healthy

```
