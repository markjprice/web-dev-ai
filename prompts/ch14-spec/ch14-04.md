> **Prompt**: Please give examples of edge cases developers often forget when writing specifications. Please explain how to identify hidden assumptions in a software specification.

An edge case is any scenario that occurs at the extreme operating limits of a product. In 2026, as systems become more interconnected through AI and third-party APIs, the "edge" has expanded from simple data validation to complex **distributed state failures**.

Identifying these early in the specification phase saves roughly **10x to 40x the cost** of fixing them after the code is written.

---

## 1. Common Edge Cases Developers Forget

### A. Temporal (Time-Based) Edge Cases
Time is the most common source of "silent" production failures.
* **The Leap Year/Day Paradox:** Logic that calculates "one year from today" often crashes on February 29th if not using library-standard date math.
* **Clock Skew:** In distributed systems (like a Django app with a separate Redis cache), the server clocks might be off by a few milliseconds, breaking "order of operations" logic.
* **The "Midnight" Race Condition:** Scheduled tasks that run at `00:00:00` may collide with daily database maintenance or log rotations.

### B. User Input & "Human" Data
* **Zero-Width Characters:** Users pasting text from Word or Slack often include hidden characters (like `\u200B`) that can break database unique constraints or URL routing.
* **The "Schunthorst" Problem (Extremely Long Strings):** Does your UI break if a user has a 100-character last name or an email address that is 254 characters long?
* **Empty States vs. Null States:** What does the dashboard look like for a user who has *joined* but has *zero* data? Beginners often forget to specify the "First-Run" experience.

### C. Network & Connectivity
* **The "Lie-Fi" Effect:** The device reports it is online, but the connection is so slow that requests time out. Does your specification define a retry strategy or a partial-loading state?
* **Idempotency on Re-submission:** If a user clicks "Submit" and the loading spinner hangs, they will click it again. Does your backend create two identical records (e.g., two charges on a credit card)?

## 2. Identifying Hidden Assumptions

Hidden assumptions are "unspoken truths" that a stakeholder or developer believes are obvious, but aren't actually documented. To find them, use the **"Deconstruction Framework."**

### Technique 1: The "Always/Never" Test
Look at every requirement and ask: *"Is this ALWAYS true, or just usually true?"*
* **Assumption:** "Users will log in with an email."
* **The Reality Check:** What if they use a "Hide My Email" relay from Apple? What if they want to change their email later? Does the system support that?

### Technique 2: The "Scale-Up/Scale-Down" Thought Experiment
* **Scale Up:** If 10,000 people do this action at the *exact same second*, what breaks? (Identifies concurrency assumptions).
* **Scale Down:** If I am the only person using this app on a 5-year-old Android phone in a basement, does it work? (Identifies hardware/bandwidth assumptions).

### Technique 3: Use the "Five Whys" for Technical Constraints
If a spec says "We must use PostgreSQL," ask why.
* **The Hidden Assumption:** "We need relational data."
* **The Discovery:** You might find the real reason is just that the lead dev is most comfortable with it, even if a Document Store (NoSQL) might actually fit the survey data better.

## 3. The "Assumption vs. Specification" Matrix

| Feature | The Hidden Assumption | The Explicit Specification |
| :--- | :--- | :--- |
| **File Uploads** | "Users will upload small JPEGs." | "System supports files up to 10MB; rejects non-image MIME types; auto-resizes to 1200px width." |
| **Notifications** | "Emails will be delivered instantly." | "Emails are queued via Celery; failed sends retry 3x; user is notified in-app if email bounces." |
| **Search** | "Users know exactly what they are looking for." | "Search supports fuzzy matching (Levenshtein distance of 2) and highlights keywords in results." |

## 4. Pro-Tip: The "Pre-Mortem" Meeting
Before finalizing a specification, hold a 15-minute "Pre-Mortem." Ask the team: **"It is six months from now, and this feature has failed catastrophically. What happened?"** This psychological shift helps developers spot edge cases like "The API key expired" or "The database grew too large for the current index" that they would otherwise ignore during the "Happy Path" design phase.
