> **Prompt**: Please explain how Spec Kit helps prevent AI coding assistants from introducing unintended features.

In the context of 2026 AI-augmented development, a **Spec Kit** is a modular bundle of machine-readable constraints (OpenAPI files, JSON Schemas, and `.prompt` files) that acts as a "logic anchor" for an AI agent. 

Without a Spec Kit, an AI coding assistant operates on **unconstrained probability**, often adding "helpful" but unnecessary features that bloat your codebase or introduce security risks. A Spec Kit shifts the AI’s goal from "write code that works" to "write code that satisfies this exact contract."

---

## 1. Establishing the "Negative Space"
AI models are trained on vast repositories of open-source code. When you ask for a feature, they tend to include everything they've ever seen associated with that feature.

* **The "Unintended" Problem:** If you ask for a "Login View," an AI might automatically add "Sign up with Twitter" or "Password Strength Meter" because those patterns are common.
* **The Spec Kit Solution:** The kit defines **Non-Goals** and strict interface boundaries. By feeding a machine-readable "Scope Manifest" into the AI's context window, you explicitly tell the model: *"If it isn't in this YAML file, do not build it."*

---

## 2. Preventing "Library Drift"
AI assistants often "hallucinate" dependencies or pull in massive third-party libraries for simple tasks (e.g., importing `Moment.js` for a simple date format in 2026).

* **Constraint Injection:** A Spec Kit includes a `dependency-allowlist.json`. 
* **Mechanism:** When the AI agent (like Claude Code or Codex) initializes, it reads this kit. If the model attempts to generate code using an unapproved library, the "Linting" layer of the Spec Kit triggers an immediate correction before the code is even presented to you.

---

## 3. Enforcement of Architectural "Seams"
One of the most common AI errors is "leaking" logic across layers—for example, putting database queries directly into a React component.



* **The Spec Kit Approach:** The kit provides **Reference Templates** for each layer (Models, Services, API). 
* **How it Works:** The AI is instructed to use the Spec Kit’s `interfaces.d.ts` (for TypeScript) or `contracts.py` (for Django) as the **only** permitted way to move data between modules. This prevents the AI from "inventing" new, non-standard ways to connect parts of your app.

---

## 4. Comparison: AI with vs. without a Spec Kit

| Feature | AI without Spec Kit | AI with Spec Kit |
| :--- | :--- | :--- |
| **Feature Set** | "Kitchen Sink" (Extra features added). | **Strict MVP** (Only what's defined). |
| **Logic Placement** | Scattered (Logic in views/templates). | **Layered** (Follows Service Pattern). |
| **Dependencies** | Random (Whatever the model likes). | **Pinned** (Uses your approved list). |
| **Security** | Generic/Assumed. | **Contract-based** (Explicitly secure). |



---

## 5. The "Verification Gate" Workflow
A Spec Kit turns the AI into a **Test-Driven Agent**. Before the AI writes the implementation, the Spec Kit forces it to generate a **Compliance Report**.

1.  **Input:** You provide a prompt + the Spec Kit.
2.  **Analysis:** The AI compares your prompt against the Spec Kit's constraints.
3.  **Conflict Check:** The AI says: *"You asked for a 'Social Share' button, but the Spec Kit lists 'No External Social APIs' as a Non-Goal. Should I proceed or skip?"*
4.  **Generation:** Only after resolving conflicts does the AI generate the code.

By using a Spec Kit, you aren't just giving the AI instructions; you are giving it a **definition of 'Done'** that it cannot ignore.
