> **Prompt**: Please explain how developers can review AI-generated code effectively.

Reviewing AI-generated code is fundamentally different from reviewing human-written code. While a human colleague might make a logical error or a typo, an AI is prone to **"confident hallucinations"**—code that looks syntactically perfect and follows every stylistic convention but fails catastrophically in edge cases or under high load.

In 2026, the industry standard is to treat AI output as **"Untrusted Input"** that must pass through a multi-stage verification pipeline.

---

## 1. The "Three-Pass" Review Framework
To avoid "Reviewer Fatigue" (where you stop paying attention because the code *looks* fine), use this structured approach:

### Pass 1: The Integrity Check (Security & Architecture)
Before looking at the logic, check the foundations.
* **Library Audit:** Did the AI import a real library or a hallucinated one? In 2026, "typosquatting" (malicious packages with names similar to real ones) is a major risk for AI-suggested dependencies.
* **Data Flow:** Does the code follow your project's architectural patterns (e.g., Service Layer vs. Fat Models)?
* **Secrets & Hardcoding:** Search for placeholder strings, mock API keys, or hardcoded IP addresses the AI might have inserted as "examples."

### Pass 2: The Logic Trace (Edge Cases)
Don't just read the code; execute it mentally using **Boundary Value Analysis**.
* **Nulls and Empties:** How does this function handle an empty list, a `None` value, or a 0? AI often ignores the "Empty State."
* **The "Happy Path" Bias:** AI models are trained on optimized examples. They often skip retry logic, timeout handling, or graceful error messages for 500-level server errors.

### Pass 3: The Performance & Style Pass
* **Complexity:** Look for $O(n^2)$ loops or "N+1" query problems in database calls.
* **Consistency:** Does the naming convention match your `constitution.md` or existing codebase?



---

## 2. Using "AI to Check AI" (The Auditor Pattern)
One of the most effective ways to review code in 2026 is to use a second, different AI model to audit the first one.

**The Prompt Strategy:**
> "I am going to provide code generated by an AI assistant. Your job is to act as a **Hostile QA Engineer**. Find three ways this code could fail in a production environment with 1 million users, specifically looking for race conditions or memory leaks."

By giving the second AI a "negative" persona, you force it to look for flaws rather than just agreeing with the first model's output.

---

## 3. Red Flags in AI Code
If you see these "smells," you should likely reject the PR or ask for a total rewrite:

| AI Code Smell | The Hidden Risk |
| :--- | :--- |
| **Over-commenting** | The AI is explaining *what* the code does because the logic itself is convoluted or non-standard. |
| **Generic Exceptions** | Using `except Exception: pass`—this hides critical failures and makes debugging impossible. |
| **Nested Ternaries** | `x if y else z if a else b`. AI loves compact syntax that is unreadable for human maintainers. |
| **Mock Data Leakage** | The function returns `{"status": "success"}` regardless of the actual database result. |

---

## 4. Automation: The First Line of Defense
Never perform a manual review before the automated tools have finished. Your **CI/CD pipeline** should act as the "gatekeeper" for AI code:
1.  **Linters & Formatters:** (e.g., Black, Ruff, ESLint) Catch stylistic issues.
2.  **Static Analysis (SAST):** (e.g., Bandit, SonarQube) Identify security vulnerabilities like SQL injection or insecure hashing.
3.  **Unit Tests with High Coverage:** If the AI didn't generate tests, make it generate them—then review the *tests* first.



### The "Senior" Mindset
A senior developer doesn't ask, *"Does this code work?"* They ask, *"Under what specific circumstances will this code break, and who will have to fix it at 3:00 AM?"*
