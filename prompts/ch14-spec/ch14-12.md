> **Prompt**: Design a specification for drag-and-drop ranking questions in a survey app and explain the data modeling challenges involved in ranking questions.

This specification defines a **Drag-and-Drop Ranking Component** for TallyApp. It allows respondents to prioritize a list of items by visually reordering them. While intuitive for users, ranking questions introduce complex data integrity and analytical challenges for developers.

---

## 1. Feature Specification: Drag-and-Drop Ranking

**User Story:**
> *As a respondent, I want to drag and drop options into my preferred order so that I can clearly communicate my priorities without typing numbers.*

### A. Functional Requirements
* **Initial State:** Options are presented in a randomized "pool" to avoid **positional bias** (the tendency to leave items in their default order).
* **Interaction:** Users click and hold a "handle" icon to move items. The interface must provide clear visual feedback (e.g., a "ghost" image of the item or a highlighted insertion gap).
* **Constraints:**
    * **Partial Ranking:** Allow admins to require a "Top 3" only, or force a full ranking of all $N$ items.
    * **Mobile-First:** Ensure the "touch" hitboxes are large enough for thumb interactions (minimum **44x44 pixels**).

### B. Interface States
1.  **Idle:** List of options with drag handles.
2.  **Dragging:** Item is lifted; remaining items shift to show the potential drop target.
3.  **Settled:** New order is locked in; index numbers ($1, 2, 3...$) update automatically.

## 2. Data Modeling Challenges

Ranking questions are significantly harder to model than single-choice questions because the "answer" is an **ordered set**, not a single value.

### Challenge 1: Representing "The Order"
If you store the rank as a simple integer on an `Option` table, you create a race condition. If two users update their order simultaneously, the integers might overlap or skip.
* **The Fix:** Store the submission as an **Ordered Array of IDs**. In PostgreSQL, this is a `uuid[]` or `integer[]` field.

### Challenge 2: Handling "Tied" or "Incomplete" Ranks
If a user is asked to rank 10 items but only ranks 3, how do you treat the remaining 7?
* **The Fix:** Use a **"Null-Last" logic**. The unranked items are treated as a tied "least preferred" group, or excluded from the mean calculation entirely.

### Challenge 3: Aggregation Logic (The "Borda Count")
You cannot simply average the ranks (e.g., "The average rank was 2.5"). 
* **The Fix:** Use the **Borda Count method**. Assign points based on position (e.g., in a list of 5, 1st place gets 5 points, 2nd gets 4, etc.). The item with the most total points is the "winner."

## 3. Implementation Design (Django)

### The Model
```python
from django.db import models
from django.contrib.postgres.fields import ArrayField

class RankingResponse(models.Model):
    submission = models.ForeignKey('Submission', on_delete=models.CASCADE)
    question = models.ForeignKey('Question', on_delete=models.CASCADE)
    
    # Stores the IDs in the order the user chose: [102, 105, 101, 104]
    ordered_item_ids = ArrayField(models.UUIDField())

    class Meta:
        constraints = [
            # Prevent duplicate rankings for the same question in one submission
            models.UniqueConstraint(fields=['submission', 'question'], name='unique_ranking')
        ]
```

### The Validation Logic
Before saving, the backend must verify:
1.  **Completeness:** Do the IDs in the array actually belong to the specific question?
2.  **Uniqueness:** Did the user try to put the same ID in two different positions (a common exploit in malformed API requests)?

## 4. Edge Cases to Watch For

* **The "Heavy" List:** If a user is asked to rank 20+ items, the DOM becomes sluggish. You must use **Virtual Scrolling** to render only the visible items.
* **Accessibility (A11y):** Drag-and-drop is impossible for screen-reader users. You **must** provide a fallback "up/down" button interface or a numeric input field hidden for sighted users but accessible to ARIA.
* **Interrupted Drag:** If the browser tab crashes mid-drag, the state should persist in `sessionStorage` so the user doesn't have to restart the 20-item ranking.
