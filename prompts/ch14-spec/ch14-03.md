> **Prompt**: Please show me some examples of poorly written acceptance criteria and how to rewrite them into clear, testable requirements.

Poorly written **Acceptance Criteria (AC)** are usually the root cause of "scope creep" and failed QA cycles. When ACs are vague, developers build what they *think* you want, and testers can’t definitively say if the feature is "done."

In 2026, with the rise of AI-driven testing tools, **testability** is the most important trait of a requirement. If a human can't test it, an AI agent certainly can't.

---

## 1. The "Subjective" Trap
Vague adjectives like "fast," "user-friendly," or "intuitive" are impossible to test because they mean different things to different people.

* **❌ Poor:** "The survey results dashboard should load fast."
* **✅ Improved:** "The survey results dashboard must have a Largest Contentful Paint (LCP) of **under 1.2 seconds** for a dataset of 10,000 responses on a 4G connection."

---

## 2. The "Negative" Requirement
Telling a developer what *not* to do without defining the expected behavior leads to "hidden" bugs.

* **❌ Poor:** "Users shouldn't be able to submit the survey twice."
* **✅ Improved:** "If a user with an existing `submission_id` in the session cookie attempts to access `/survey/submit/`, the system must redirect them to `/survey/already-submitted/` and display a message: 'You have already completed this survey.'"

---

## 3. The "Multi-Requirement" AC (The Blob)
Cramming five different behaviors into one bullet point makes it impossible to track progress. If 4 out of 5 parts work, is the ticket "Done"?

* **❌ Poor:** "The app should handle login, password resets, and social auth via Google and GitHub."
* **✅ Improved (Break into separate points):**
    * "Users can authenticate using a valid Email and Password combination."
    * "Users can trigger a 'Forgot Password' flow that sends a unique, 15-minute expiration token via SendGrid."
    * "Users can sign in via the OpenID Connect (OIDC) flow using a verified Google account."

---

## 4. The "Missing State" AC
Beginners often write ACs for the "Happy Path" but forget about errors, empty states, or loading states.

* **❌ Poor:** "Show the list of survey responses."
* **✅ Improved:**
    * "When 1 or more responses exist, display them in a paginated table (25 per page) sorted by `created_at` descending."
    * "If 0 responses exist, display the `empty-state.svg` illustration with a 'Share Survey' Call-to-Action button."

---

## Comparison: From Vague to Professional

| Feature | Poorly Written AC | Clear & Testable AC |
| :--- | :--- | :--- |
| **Search** | "Search should be accurate." | "Searching for a partial string (e.g., 'Tal') must return all `Survey` titles containing that string, case-insensitive." |
| **Security** | "The API must be secure." | "All requests to `/api/v1/private/` must return a `401 Unauthorized` status code if the `Authorization` header is missing or invalid." |
| **UI/UX** | "The button should look good." | "The 'Submit' button must use the primary brand hex color `#3B82F6` and change to `#2563EB` on hover." |

---

## The "Given-When-Then" Framework
A gold standard for writing clear ACs is the **Gherkin Style**. This format turns a requirement into a literal test script.

> **Scenario:** Successful Survey Submission
> * **Given** a user has filled out all required fields in the 'Employee Pulse' survey.
> * **When** the user clicks the 'Submit' button.
> * **Then** the system creates a new record in the `Responses` table.
> * **And** the user is redirected to a 'Thank You' page.



---

## Why This Matters for AI Collaboration
If you provide the "Poor" examples to an AI coding assistant, it will hallucinate the missing details. If you provide the "Improved" examples, the AI will generate the exact logic, validation, and unit tests you actually need.
