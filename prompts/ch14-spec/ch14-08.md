> **Prompt**: Please explain how GitHub Spec Kit works internally and what files it generates in a project repository. Show the directory structure created by Spec Kit and explain the purpose of each file.

In the 2026 development landscape, **GitHub Spec Kit** has become the standard for **Spec-Driven Development (SDD)**. Internally, it operates as a "governance layer" that sits between your natural language intent and the AI coding agent (like Copilot, Claude Code, or Gemini).

It works by transforming your project into a **state machine**. Instead of the AI guessing what to do next ("vibe coding"), Spec Kit forces the agent to transition through explicit, gated phases: **Specify $\rightarrow$ Plan $\rightarrow$ Tasks $\rightarrow$ Implement.**

---

## 1. Directory Structure Created by Spec Kit
When you initialize a project using the `specify init` command (via the CLI), Spec Kit scaffolds a hidden `.specify/` directory. This folder acts as the "source of truth" for the AI's memory and the project's rules.

```text
my-django-app/
├── .specify/                # The core SDD engine
│   ├── memory/
│   │   └── constitution.md  # Project principles & guardrails
│   ├── scripts/             # Automation for the SDD workflow
│   │   ├── common.sh
│   │   ├── setup-plan.sh
│   │   └── update-agent-context.sh
│   ├── specs/               # High-level feature definitions
│   │   └── 001-user-auth.md
│   ├── plans/                # Technical implementation details
│   │   └── 001-user-auth-plan.md
│   └── templates/           # Blueprints for new specs/plans
│       ├── spec-template.md
│       └── plan-template.md
├── .github/                 # Agent-specific instructions
│   └── prompts/
│       ├── specify.md       # Logic for the /specify command
│       └── plan.md          # Logic for the /plan command
└── SPEC.md                  # The active "living" specification
```

---

## 2. Purpose of Key Files

### `.specify/memory/constitution.md`
This is arguably the most important file. It contains the **"Non-Negotiables"** for your project. If you tell the constitution "Never use Inline CSS" or "Always use Django Ninja for APIs," the AI agent is mathematically weighted to follow these rules during the `/plan` phase.

### `.specify/specs/` and `SPEC.md`
These files store the "What" and "Why." They describe user scenarios, motivations, and **Acceptance Criteria**. Unlike a static PDF, these are living Markdown files that the AI reads before every coding task to ensure alignment with your original intent.

### `.specify/plans/`
The "Plan" is the bridge between a business requirement and code. It defines the **Stack Choices** (e.g., "Use PostgreSQL 16," "Use Tailwind CSS"). By recording these in a file, you prevent the AI from "hallucinating" a different tech stack halfway through the project.

### `.specify/scripts/`
These are bash/PowerShell scripts that automate context management. For example, `update-agent-context.sh` automatically gathers the relevant bits of your `constitution.md` and `SPEC.md` and feeds them into the AI's prompt so you don't have to copy-paste them manually.

---

## 3. How it Prevents "Unintended Features"
Spec Kit uses a **contract-first** approach. When you run the `/tasks` command, Spec Kit scans your `SPEC.md` and `PLAN.md` to generate a backlog.

1.  **Scope Anchoring:** The AI can only generate tasks that have a direct "parent" requirement in the Spec.
2.  **Gated Implementation:** The `/implement` command is typically locked until the `/plan` has been reviewed and "signed off" (merged) by a human.
3.  **Context Hygiene:** By using the scripts in `.specify/`, the toolkit ensures the AI only sees the *relevant* requirements for the current task, reducing the noise that often leads to AI "hallucinations" or over-engineering.
