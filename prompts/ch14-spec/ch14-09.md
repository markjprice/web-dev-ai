> **Prompt**: How should developers structure prompts when asking AI to generate an implementation plan?

When asking an AI to generate an implementation plan, you are moving from the **"What"** (the requirements) to the **"How"** (the technical execution). In 2026, the gold standard for this is the **Spec-Driven Development (SDD)** approach used by tools like **GitHub Spec Kit**.

The key is to treat the AI as a **Senior Technical Architect**. You aren't just asking for code; you are asking for a structured roadmap that respects your existing constraints and architectural principles.

---

## 1. The Implementation Plan Prompt Framework
A professional implementation plan prompt should follow the **P.S.A.L.M.** structure: **Persona, Source, Architecture, Logic, and Milestone.**

> **1. Persona:** Set the AI's role (e.g., "Act as a Senior Django Architect").
> **2. Source:** Reference your **Single Source of Truth** (the PRD or `SPEC.md`).
> **3. Architecture:** Define the tech stack, libraries, and design patterns.
> **4. Logic:** Specify how data flows and how edge cases are handled.
> **5. Milestones:** Ask for a phased breakdown (Phase 0: Research, Phase 1: Models, etc.).

---

## 2. Example: Bad vs. Professional Planning Prompt

### ❌ The "Vague" Prompt
> "Write a plan to build the survey submission feature for my Django app."
* **Result:** The AI will guess your database schema, ignore your authentication logic, and likely suggest a monolithic structure that doesn't scale.

### ✅ The "Spec-Driven" Prompt (Using 2026 Standards)
> "Act as a Senior Backend Engineer. Based on the attached `SPEC.md` for the TallyApp submission engine, generate a detailed **Technical Implementation Plan**.
>
> **Constraints:**
> * **Stack:** Django 5.2, PostgreSQL (JSONB for answers), Redis for caching.
> * **Architecture:** Use a **Service Layer pattern**; no business logic in `views.py`.
> * **Security:** Ensure all submissions are validated via a Pydantic schema before hitting the ORM.
> 
> **Required Output Structure:**
> 1. **Technical Context:** List all required models and third-party dependencies.
> 2. **Constitution Check:** Verify this plan doesn't violate our project's 'No IP Logging' rule.
> 3. **Execution Phases:** Break the work into 4 distinct phases (Data Model, API Contracts, Logic, and Testing).
> 4. **Unknowns:** Flag any parts of the `SPEC.md` that are too ambiguous to implement."

---

## 3. Advanced Strategy: The "Constitution" Check
In modern workflows, the implementation plan is often checked against a **Project Constitution**. This is a file (like `.specify/memory/constitution.md`) that contains your team's hard rules.

**Add this to your prompt:**
> "Before finalizing the plan, run a 'Constitution Check.' Identify any areas where this implementation might conflict with our project's performance standards (e.g., 'All API responses must be <200ms') or security guardrails."



---

## 4. Why This Structure Reduces "Plan Drift"
By forcing the AI to list **Unknowns** and **Dependencies** before writing tasks, you prevent "Plan Drift"—where the AI starts building a feature based on a wrong assumption.

| Prompt Component | Purpose |
| :--- | :--- |
| **"NEEDS CLARIFICATION"** | Forces the AI to admit it doesn't understand a requirement rather than guessing. |
| **Data Model Extraction** | Ensures the AI maps out the SQL/NoSQL structure before the API logic. |
| **Interface Contracts** | Defines exactly what the inputs and outputs look like (e.g., OpenAPI schemas). |

### Pro-Tip: The "Phase 0" Research
If your plan involves a new library (like a specific 2026 encryption tool), ask the AI to include a **"Phase 0: Research"** task. 
* *Example:* "Include a research task to verify if `LibraryX` is compatible with our current Python 3.14 environment."
