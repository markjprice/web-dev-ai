> **Prompt**: Design a specification for adding branching logic to a survey system.

This specification defines the **Conditional Branching Engine** for TallyApp. Branching logic (also known as "Skip Logic") allows a survey to adapt in real-time based on a respondent's previous answers, ensuring users only see questions relevant to them.

---

## 1. Feature Specification: Branching Logic

**User Story:**
> *As a survey creator, I want to show or hide specific questions based on the respondent's previous answers so that I can reduce survey fatigue and collect cleaner data.*

### A. Functional Requirements
* **Rule Types:**
    * **Question-to-Question:** If Question A is "Yes," show Question B.
    * **Question-to-Page:** If Question A is "No," skip to Page 4.
    * **Terminal Logic:** If Question A is "Ineligible," end the survey immediately with a custom "Disqualified" message.
* **Logical Operators:** Support `AND`, `OR`, and `NOT` (e.g., "Show Question 5 if Q1 is 'Yes' AND Q2 is > 18").
* **Visibility Toggle:** Questions hidden by logic must have their values cleared or ignored to prevent "stale" data from being submitted.



---

## 2. Technical Architecture & Data Modeling

The primary challenge with branching is representing a **Directed Acyclic Graph (DAG)** within a relational database.

### A. The "Rule" Entity Model
Instead of hardcoding logic into the `Question` model, we use a separate `LogicRule` table to maintain a clean separation of concerns.

```python
from django.db import models

class LogicRule(models.Model):
    # The question that "owns" the logic (the target)
    target_question = models.ForeignKey('Question', related_name='visibility_rules', on_delete=models.CASCADE)
    
    # The condition that must be met
    source_question = models.ForeignKey('Question', related_name='triggered_rules', on_delete=models.CASCADE)
    operator = models.CharField(max_length=20, choices=[
        ('equals', 'Equals'),
        ('not_equals', 'Does Not Equal'),
        ('greater_than', 'Greater Than'),
        ('contains', 'Contains Any Of'),
    ])
    
    # The comparison value stored as JSON to handle strings, numbers, or lists
    value = models.JSONField()
    
    # Action to take
    action = models.CharField(max_length=20, choices=[
        ('show', 'Show Question'),
        ('hide', 'Hide Question'),
        ('jump_to', 'Jump to Page'),
    ])
```

### B. Execution Philosophy: Frontend vs. Backend
* **Frontend (Real-time):** A JavaScript "Evaluator" runs on every input change. It scans the `LogicRules` for the current page and toggles CSS visibility or DOM presence.
* **Backend (Validation):** When the survey is submitted, the server **must** re-evaluate the logic. If a user "hacked" the CSS to answer a hidden question, the backend must discard that data to maintain integrity.

## 3. Critical Challenges & Edge Cases

### Challenge 1: The "Infinite Loop"
A beginner mistake is allowing circular logic (e.g., Q1 depends on Q2, and Q2 depends on Q1).
* **The Fix:** The UI must perform a **Cyclic Dependency Check** whenever a rule is saved. In a Django backend, you can use a depth-first search (DFS) algorithm to ensure the logic graph is acyclic.

### Challenge 2: "Backward" Logic
What happens if a user answers Q1, sees Q2, answers Q2, and then goes back to change Q1?
* **The Fix:** If the "Source" question changes and the condition is no longer met, all "downstream" answers must be **recursively reset**.

## 4. Implementation Checklist for 2026

| Component | Responsibility | Strategy |
| :--- | :--- | :--- |
| **Schema** | Backend (Django) | Use `JSONField` for flexible comparison values. |
| **API** | REST/GraphQL | Send all logic rules for the *current page* in the initial page load to minimize latency. |
| **State** | Frontend (React/Vue) | Use a global state manager (Redux/Pinia) to track "Visibility Map" of all IDs. |
| **UX** | Designer | Use "Connector Lines" in the admin builder to visually show which questions are linked. |

### Non-Goals for MVP
* **External API Logic:** We will not support rules based on external weather APIs or stock prices in Version 1.
* **Cross-Survey Logic:** Logic cannot depend on answers given in a *different* survey.
