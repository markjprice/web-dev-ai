> **Prompt**: Design a specification for geographic survey responses using a map interface.

This specification defines a **Geographic Survey Component** for TallyApp. It allows users to submit responses by interacting with a map (dropping pins, drawing areas) and ensures the resulting data is structured for high-performance spatial analysis in a Django/PostgreSQL (PostGIS) backend.

---

## 1. Feature Overview
The Geographic Survey interface enables users to provide location-based data. Instead of typing an address, users interact with an embedded map to define points, lines, or boundaries.

**Primary User Story:**
> *As a respondent, I want to select my exact location on a map or outline a specific neighborhood so that my feedback is geographically accurate.*

---

## 2. Technical Specifications

### A. Interface Requirements (Frontend)
The map interface should be powered by **MapLibre GL JS** or **Leaflet** using vector tiles for performance.

* **Interaction Modes:**
    * **Point:** Drop a single pin.
    * **Radius:** Drop a pin and define a buffer (e.g., "within 500m").
    * **Polygon:** Draw a custom boundary (for neighborhood or zone feedback).
* **Geocoding:** A search bar to fly to a specific address before fine-tuning the pin.
* **Metadata Capture:** Automatically capture the zoom level and map style (Satellite vs. Street) used during submission to understand user context.



### B. Data Schema (Backend)
To support 1 million+ responses (as discussed in your scaling query), we use **GeoJSON** format stored in a PostGIS-enabled `GeometryField`.

| Field Name | Data Type | Description |
| :--- | :--- | :--- |
| `location_type` | `Enum` | 'Point', 'Polygon', or 'Circle'. |
| `geometry` | `GeometryField` | Standard SRID 4326 (WGS 84) spatial data. |
| `accuracy_meters` | `FloatField` | The estimated margin of error from the device GPS. |
| `bounds` | `Box2D` | The bounding box of the user's view during submission. |

## 3. Implementation Plan (Django + PostGIS)

### Phase 1: Models & Spatial Indexing
```python
from django.contrib.gis.db import models

class GeoResponse(models.Model):
    submission = models.OneToOneField('Submission', on_delete=models.CASCADE)
    # Supports points or areas
    geom = models.GeometryField(srid=4326, spatial_index=True) 
    
    class Meta:
        indexes = [
            # Spatial index for lightning-fast proximity queries
            models.Index(fields=['geom'], name='spatial_idx')
        ]
```

### Phase 2: Logic & Validation
* **Coordinate Integrity:** Reject any coordinates that fall outside the "Survey Bounds" defined by the admin.
* **Simplification:** If a user draws a complex polygon with 500 points, use the **Douglas-Peucker algorithm** to simplify the geometry before saving to reduce database bloat.

---

## 4. Edge Cases & Non-Goals

### Edge Cases to Handle:
* **Offline Submission:** If the user drops a pin while offline, store the raw lat/lng in `localStorage` and sync when the connection returns.
* **Ambiguous Centroids:** If a user draws a "U-shaped" polygon, ensure the label/pin logic doesn't place the center in "empty space" outside the shape.
* **High Density:** If 10,000 users drop pins in the same city block, the admin dashboard must use **Marker Clustering**.



### Non-Goals:
* **Real-time Tracking:** This is a one-time submission tool, not a live GPS tracker.
* **3D Modeling:** Support for Z-axis (altitude) or 3D terrain drawing is out of scope for the MVP.

---

## 5. Security & Privacy (The "Anonymity" Constraint)
Since this is an **Anonymous Survey**, geographic data is a major deanonymization risk (the "Home Address" problem).

* **Privacy Fuzzing:** If the survey is sensitive, the system should optionally "jitter" the coordinates by a random offset (e.g., 50–100m) or snap them to the nearest neighborhood centroid to protect the user's exact house location.
* **Strict Access Control:** Spatial data should be encrypted at rest and only decrypted when generating aggregate heatmaps.
