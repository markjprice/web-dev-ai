> **Prompt**: Please explain how URL shorteners work internally. Describe the algorithms commonly used to generate short URL codes and compare random code generation with hash-based URL shortening.

At its core, a URL shortener is a **high-performance lookup table** that maps a long, complex URL (the "Target") to a short, unique string (the "Key"). When a user visits `short.ly/xyz123`, the server looks up `xyz123` in its database and issues an **HTTP 301 (Permanent)** or **302 (Temporary) Redirect** to the destination.

---

## 1. Internal Architecture & Data Flow

A URL shortener must handle two distinct operations: **Shortening** (Write) and **Redirection** (Read). Because redirection happens much more frequently than creation, the system is typically optimized for read-heavy workloads using a caching layer.

1.  **Request:** The user provides a long URL.
2.  **Validation:** The system checks if the URL is valid and safe (not a phishing site).
3.  **Key Generation:** An algorithm creates a unique short code (e.g., `5aZ7r`).
4.  **Storage:** The mapping `{short_code: long_url}` is saved to a database (usually a NoSQL store like Redis or Cassandra for speed).
5.  **Retrieval:** When the short URL is clicked, the system fetches the long URL from the cache/DB and redirects the user.



---

## 2. Algorithms for Short Code Generation

The goal is to represent a large integer (the database ID) using a small number of characters. This is almost universally done using **Base 62 Encoding**.

### Base 62 Encoding
Standard decimal is Base 10 ($0-9$). Hexadecimal is Base 16 ($0-9, A-F$). Base 62 uses:
* $0-9$ (10 characters)
* $a-z$ (26 characters)
* $A-Z$ (26 characters)

**The Math:** A 6-character code in Base 62 provides $62^6$ unique combinations—roughly **56.8 billion** possible URLs. 

---

## 3. Random Generation vs. Hash-Based Shortening

There are two primary philosophies for generating that unique "Key."

### A. Counter-Based (Sequential) with Base 62
The system maintains a global counter (e.g., in a Distributed ID Generator like **Snowflake**). 
* **Process:** Take the ID (e.g., `10,001`), convert it to Base 62 (`2Bi`), and use that as the key.
* **Pros:** Guaranteed uniqueness; very fast.
* **Cons:** Predictable. Anyone can guess the "next" URL by incrementing the code (e.g., `2Bj`).

### B. Hash-Based Shortening
The system passes the long URL through a hashing algorithm like **MD5** or **SHA-256**.
* **Process:** Hash the URL $\rightarrow$ Take the first 7 characters of the hash $\rightarrow$ Use as the key.
* **Pros:** The short code is directly tied to the content; same URL always gets the same code.
* **Cons:** **Collisions.** Two different URLs might produce the same first 7 characters. You must handle this by appending a "salt" or timestamp and re-hashing if a collision occurs.



---

## 4. Comparison Table

| Feature | Random/Counter-Based | Hash-Based |
| :--- | :--- | :--- |
| **Logic** | Convert ID to Base 62. | Hash URL and truncate. |
| **Uniqueness** | Guaranteed by the ID counter. | Risk of collisions. |
| **Predictability** | High (can be "crawled"). | Low (looks random). |
| **Performance** | Extremely fast. | Slower (requires hashing + collision checks). |
| **Idempotency** | Creates new links for same URL. | Can return same link for same URL. |

---

## 5. Which is Better?
In modern distributed systems (like **Bitly** or **TinyURL**), the **Counter-Based** approach is generally preferred for its simplicity and speed. To solve the "predictability" problem, engineers often **shuffle** the Base 62 alphabet or use a "Feistel Cipher" to encrypt the sequential ID so it *looks* random to the end user while remaining mathematically unique.
