> **Prompt**: What deprecations or breaking changes should I check before upgrading my project to the latest TypeScript version?

Upgrading your project to the latest versions (currently **TypeScript 6.0 stable** and **7.0 Beta**) involves significant architectural and configuration shifts. Because TypeScript 6.0 and 7.0 (the Go-based "Native" version) aim to modernize the ecosystem, they remove many legacy options that were previously just "discouraged."

Below is a breakdown of the critical deprecations and breaking changes you should verify.

## 1. Major Configuration & CLI Changes
TypeScript 6.0 changed many long-standing defaults. If your `tsconfig.json` relies on "old school" behavior, you may encounter immediate build errors.

| Feature | Change in 6.0 / 7.0 | Recommended Action |
| :--- | :--- | :--- |
| **`strict`** | Now defaults to `true`. | Explicitly set `strict: false` if your project isn't ready. |
| **`moduleResolution`** | `node` (Node10) is deprecated. | Migrate to `nodenext` or `bundler`. |
| **`target`** | `ES5` is deprecated; defaults to `ES2025`. | Minimum target is now **ES2015**. |
| **`baseUrl`** | Deprecated for general lookups. | Use **`paths`** for explicit module mapping. |
| **`outFile`** | Removed. | Use a modern bundler (Vite, esbuild, Webpack). |
| **`imports`** | `asserts` keyword is removed. | Replace with `with` (e.g., `import json from "./f.json" with { type: "json" }`). |

---

## 2. The Move to "Erasable Syntax"
With the introduction of native execution in Node.js and the performance-focused **TypeScript 7.0 (Native)**, there is a push to avoid syntax that requires a complex runtime transformation.

* **`--erasableSyntaxOnly`**: This new flag (introduced in 5.8/6.0) will error on constructs like **Enums**, **Namespaces**, and **Parameter Properties** (e.g., `constructor(public x: number)`). 
* **Recommendation**: If you plan to move to the 10x faster **tsgo** compiler in 7.0, start refactoring Enums into `const` objects or literal unions.

---

## 3. Breaking Type System Changes


### Strict Generic Inference (`--strictInference`)
In version 5.9 and 6.0, TypeScript became stricter about how it infers types for generics. 
* **The Change**: If the compiler cannot confidently infer a type, it no longer falls back to `unknown` or the constraint silently; it now throws an error.
* **The Fix**: You must provide explicit type arguments in complex generic calls where inference is ambiguous.

### `lib.dom.d.ts` Updates
As always, the DOM types have been updated to match 2026 web standards. 
* **Impact**: You may see errors in code interacting with `Media` APIs, `WebGPU`, or `CSS Typed OM` if your manual types conflict with the updated global definitions.

### TypedArrays are now Generic
Starting in 5.7+, `Uint8Array` and other typed arrays are generic over `ArrayBufferLike`. 
* **The Change**: `interface Uint8Array<TArrayBuffer extends ArrayBufferLike = ArrayBufferLike>`.
* **Impact**: If you have custom interfaces extending `Uint8Array`, they may need to be updated to account for the new type parameter.

---

## 4. Upgrading to TypeScript 7.0 Beta (Native)
If you are experimenting with the Go-based 7.0 release:
* **Entry Point**: You use the **`tsgo`** executable instead of `tsc`.
* **Feature Parity**: While it aims for 100% semantic parity with 6.0, it is strictly focused on modern modules. Legacy features like `namespaces` and `amd`/`systemjs` module targets are not supported in the native compiler.

## Quick Upgrade Checklist
1. [ ] Search for `import ... assert { type: 'json' }` and change to `with`.
2. [ ] Check `tsconfig.json` for `moduleResolution: "node"`; change to `"bundler"`.
3. [ ] If using Enums, consider enabling `erasableSyntaxOnly` to see how much work a "native" upgrade would take.
4. [ ] Run `npx tsc --version` to ensure you are at least on **6.0.x** before jumping to the **7.0** preview.
