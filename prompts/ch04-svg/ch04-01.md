> **Prompt**: Please explain the difference between inline SVG and <img> SVG with pros, cons, and real use cases.

When you use Scalable Vector Graphics (SVG) on a website, you generally choose between two methods: embedding the raw code directly into your HTML (**Inline**) or linking to an `.svg` file via an image tag (**`<img>`**).

The choice depends on whether you need to manipulate the image with code or simply display it as a static asset.

---

## 1. Inline SVG
This involves pasting the `<svg>` tags and all their internal paths directly into your HTML file.

### **Pros**
* **CSS Styling:** You can change colors (`fill`), strokes, and sizes using your external CSS file or hover states.
* **Interactivity:** You can target specific parts of the SVG (like a single gear in a machine icon) with JavaScript or CSS animations.
* **No Extra HTTP Request:** Since the code is in the HTML, the browser doesn't have to go fetch another file from the server.
* **Accessibility:** You can add `<title>` and `<desc>` tags inside the SVG for better screen reader support.

### **Cons**
* **Bloat:** Large, complex SVGs (like detailed illustrations) can make your HTML file massive and hard to read.
* **No Caching:** Because the SVG is part of the HTML, the browser cannot cache the image separately. If you use the same logo on 10 pages, the user downloads that code 10 times.

**Real Use Case:** UI icons that need to change color on hover or complex animations where individual parts of the graphic move independently.

---

## 2. `<img>` Tag SVG
This works exactly like a standard `.jpg` or `.png`. You host the file and link to it.
` <img src="logo.svg" alt="Company Logo"> `

### **Pros**
* **Clean Code:** Keeps your HTML file small and organized.
* **Browser Caching:** The browser downloads the file once and reuses it from the cache on every other page, saving data.
* **Ease of Use:** It’s simple to implement and works perfectly with standard responsive techniques (like `max-width: 100%`).

### **Cons**
* **Limited Styling:** You **cannot** change the internal colors or paths of the SVG using CSS from your main stylesheet.
* **No Scripting:** For security reasons, browsers disable any JavaScript or interactions embedded inside an SVG when loaded via an `<img>` tag.

**Real Use Case:** Static logos, illustrations in blog posts, or decorative background elements that don't need to change based on user interaction.

---

## Comparison Table

| Feature | Inline SVG | `<img>` Tag SVG |
| :--- | :--- | :--- |
| **CSS Control** | Full (Target individual paths) | None (Only filters like `blur`) |
| **JS Interaction** | Full | None |
| **Caching** | No | Yes |
| **HTTP Requests** | 0 | 1 |
| **Best For** | Icons, UI, Animations | Logos, Static Illustrations |

## Visual Example of SVG Structure
Here is how an inline SVG looks as raw source code. Notice how it is just a series of coordinate instructions:

```svg
<svg width="100" height="100" viewBox="0 0 100 100" xmlns="http://www.w3.org/2000/svg">
  <circle cx="50" cy="50" r="40" stroke="black" stroke-width="3" fill="red" />
  <text x="50" y="55" font-size="12" text-anchor="middle" fill="white">SVG</text>
</svg>
```



---

## Which one should you use?

* **Use Inline** if the SVG is a **functional** part of the UI (e.g., an icon that turns blue when a button is clicked).
* **Use `<img>`** if the SVG is an **asset** (e.g., a header illustration or a company logo in the footer).
