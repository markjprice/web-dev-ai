> **Prompt**: How would you modify the TallyApp one-shot prompt to enforce stricter architectural constraints?

To enforce stricter architectural constraints in a one-shot prompt, you must shift the AI's focus from "generating features" to "adhering to a structural contract." In a Django context, this means explicitly defining the **Data Flow**, **Separation of Concerns**, and **Validation Layer** before a single line of logic is written.

The most effective way to do this is to include a **"Constraint Manifesto"** within your prompt.

---

## 1. The "Architectural Guardrail" Prompt Template
To move beyond a basic prompt, add a dedicated "Technical Constraints" section. Here is how you would modify the previous Tally.so example for a Django-backed survey engine:

> **Role:** Senior Django Architect.
> **Task:** Generate a survey submission engine for a Tally-like clone.
> **Architectural Constraints:**
> 1. **Pattern:** Use the **Service Layer pattern**. Do not put business logic in `models.py` or `views.py`. Create a `services.py` for all data mutations.
> 2. **Type Safety:** Use **Python Type Hits** for all function signatures.
> 3. **Validation:** All incoming JSON must be validated via a **Pydantic schema** before reaching the ORM.
> 4. **Database:** Use `JSONField` for dynamic survey responses, but enforce a strict schema check against a `SurveyTemplate` model.
> 5. **Anonymity:** Implement a middleware constraint that strips `META['REMOTE_ADDR']` before the request reaches the view.
>
> **Output Requirement:** Before providing code, list the file structure and explain how data flows from the Request to the Database.

---

## 2. Why These Modifications Work
By adding these specific headers, you are overriding the AI's tendency to provide the "simplest" (and often messiest) solution.

### The Service Layer Requirement
Standard AI output puts logic in the `save()` method of a model. By demanding a `services.py`, you force the AI to decouple the database from the business logic, which is essential for testing and scaling.



### Schema-First Development
Forcing the use of **Pydantic** or **Django Rest Framework Serializers** prevents the AI from writing "blind" `request.POST.get()` calls, which are a primary source of security vulnerabilities and runtime crashes.

## 3. Comparison of Output Quality

| Constraint Type | Standard One-Shot Result | Modified "Strict" Result |
| :--- | :--- | :--- |
| **Logic Placement** | Fat Models / Logic in Views | **Slim Views / Dedicated Service Layer** |
| **Data Integrity** | Basic `if/else` checks | **Pydantic/Serializer Validation** |
| **Error Handling** | Generic `try/except` | **Custom Exception Classes** |
| **Testing** | Usually omitted | **Mock-ready, decoupled functions** |

---

## 4. Strategic "Negative Constraints"
Sometimes what you *don't* want is more important than what you do. Adding a "Negative Constraints" block prevents the AI from taking shortcuts:

* **"No Signals:"** Prevent the AI from using Django Signals (which are hard to debug).
* **"No Raw SQL:"** Force the use of the ORM or QuerySets to prevent injection.
* **"No Third-Party Auth:"** If you want a custom implementation, explicitly forbid `django-allauth`.

### The "System Prompt" Trick
If you are using a tool like Claude or GPT-4o, you can set these architectural constraints as a **System Prompt**. This ensures that every subsequent "shot" in an iterative process adheres to these rules without you having to repeat them.
