How would a senior engineer review AI-generated Django models?

In a tool like TallyApp, where the user interface is as simple as a text document but the data captured is highly dynamic, **modeling the answers** is the single most critical architectural decision. It dictates your app’s ability to scale, its query performance, and how easily you can build analytics dashboards.

Get this wrong, and you'll end up with a database that is either too rigid to support new question types or so "loose" that generating a simple bar chart of results takes minutes of processing time.

---

## 1. Why Answer Modeling is "The Big One"
In a standard Django app, you usually know your schema (e.g., a `Post` always has a `title` and `content`). In a survey app, the "schema" is defined by the end-user at runtime. You have to store data that could be a string, an integer, a list of IDs, or even a file path—all within a system that must remain searchable and aggregatable.

---

## 2. Three Database Designs for Survey Answers

A senior engineer chooses a design based on the **Scale vs. Flexibility** trade-off.

### Design A: The "Normalized" Relational Model
This is the "Clean Code" approach. Every answer is a row in a table, linked to a specific question and option.

```python
class Answer(models.Model):
    submission = models.ForeignKey(Submission, on_delete=models.CASCADE)
    question = models.ForeignKey(Question, on_delete=models.CASCADE)
    # Different fields for different types
    text_value = models.TextField(null=True, blank=True)
    selected_options = models.ManyToManyField('Option', blank=True) 
```

* **Single-choice:** Link to one `Option` in `selected_options`.
* **Multi-choice:** Link to multiple `Option` records.
* **Text:** Store in `text_value`.
* **Pros:** Strict data integrity; easy to count votes using SQL `COUNT` and `GROUP BY`.
* **Cons:** "Sparse" tables (many nulls); expensive joins when exporting a 50-question survey.

### Design B: The EAV (Entity-Attribute-Value) Model
This treats the database like a long list of key-value pairs.

| SubmissionID | QuestionID | Value |
| :--- | :--- | :--- |
| 101 | 1 (Color) | "Blue" |
| 101 | 2 (Features) | "[\"Fast\", \"Cheap\"]" |

* **Pros:** Infinite flexibility. You can add a "Date" or "Signature" question type without changing the schema.
* **Cons:** Loses database-level type safety. All values are usually stored as `TEXT`, requiring manual casting in Python (e.g., `int(value)`), which is slow and error-prone.

### Design C: The Modern JSONB Approach (Recommended for 2026)
This uses PostgreSQL's `JSONB` field to store the entire submission as a structured object.

```python
class Submission(models.Model):
    survey = models.ForeignKey(Survey, on_delete=models.CASCADE)
    data = models.JSONField() # { "q1": "Blue", "q2": [1, 4, 5], "q3": "Great app!" }
```

* **Pros:** Fast writes; incredibly easy to handle complex nested data (like Tally’s matrix questions).
* **Cons:** Harder to perform complex aggregations across millions of rows compared to normalized SQL; requires careful schema validation in your Python code.

## 3. Comparison of Trade-offs

| Strategy | Flexibility | Query Speed (Aggregates) | Data Integrity |
| :--- | :--- | :--- | :--- |
| **Normalized** | Low | **High** | **Highest** |
| **EAV** | **Highest** | Low | Low |
| **JSONB** | High | Moderate | Moderate |

## 4. How a Senior Engineer Reviews AI Models
When a Senior Engineer reviews Django models generated by an AI, they aren't looking for syntax errors—they are looking for **architectural foresight**.

### The "Senior Review" Checklist:
1.  **Index Optimization:** Did the AI add `db_index=True` to the `SurveyID` and `UserID`? Without these, the app will crawl once you hit 10,000 submissions.
2.  **On_Delete Logic:** AI often defaults to `models.CASCADE`. A senior will ask: *"If a user deletes a question, do we really want to wipe out 50,000 historical answers?"* They might suggest `SET_NULL` or a "Soft Delete" instead.
3.  **Validation Placement:** They check if validation is in the `clean()` method. They'll look for `JSONField` models and demand a **JSON Schema** validator to ensure the AI didn't create a "data graveyard" where any junk text can be saved.
4.  **Database Constraints:** They look for `UniqueTogether` constraints (e.g., ensuring a user can't submit the same survey twice if that’s a requirement).
5.  **The "Future" Test:** They ask, *"How does this model handle a question type the AI didn't think of, like a file upload or a ranking scale?"*

> **Senior Quote:** "AI writes code that works for the developer. Senior engineers write code that works for the database administrator three years from now."
