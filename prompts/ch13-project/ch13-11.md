> **Prompt**: How do environment variables improve E2E test reliability? Please explain how test.skip() works in Playwright and when it should be used.

In modern End-to-End (E2E) testing, **environment variables** act as the "control center," allowing your test suite to adapt to different stages (local, staging, production) without changing a single line of code. Meanwhile, tools like `test.skip()` in Playwright provide the surgical precision needed to manage a suite when conditions aren't perfect.

---

## 1. How Environment Variables Improve E2E Reliability

Reliability in E2E testing often comes down to **Environment Parity** and **Secret Management**. Environment variables (`.env` files or CI/CD secrets) provide three major reliability boosts:

### A. Dynamic Configuration (The "Single Source of Truth")
Hardcoding URLs or credentials is the fastest way to "flaky" tests. If a developer changes a staging URL, a hardcoded suite breaks. With environment variables, your code stays static:
* **Base URLs:** `page.goto(process.env.BASE_URL)` ensures tests always run against the correct target.
* **Feature Flags:** You can enable or disable tests based on whether a feature is "on" in a specific environment.

### B. Security & Clean State
Reliability isn't just about passing tests; it's about predictable data. 
* **Authentication:** Using `process.env.TEST_USER` allows you to rotate credentials globally without touching the test scripts.
* **Database Isolation:** You can point your tests to a dedicated "Test DB" via environment variables to ensure test data doesn't pollute production or conflict with other developers.

### C. CI/CD Portability
By injecting variables through your CI pipeline (GitHub Actions, GitLab CI), you ensure the tests run exactly the same way in the cloud as they do on your laptop, eliminating "it works on my machine" syndrome.

## 2. How `test.skip()` Works in Playwright

In Playwright, `test.skip()` is a directive that tells the test runner to **ignore** a specific test or a group of tests entirely. When a test is skipped, it is not executed, and it is marked as "skipped" in the final report.

### Syntax Examples
* **Single Test:**
    ```javascript
    test.skip('hidden feature test', async ({ page }) => {
      // This code will not run
    });
    ```
* **Conditional Skipping (The "Power Move"):**
    ```javascript
    test('complex dashboard', async ({ page }, testInfo) => {
      // Skip only on WebKit (Safari) because of a known bug
      test.skip(testInfo.project.name === 'webkit', 'Still working on Safari fix');
      
      await page.goto('/dashboard');
    });
    ```

---

## 3. When to Use `test.skip()` (and When Not To)

`test.skip()` should be used as a **temporary management tool**, not a permanent solution for broken code.

| Best Use Case | Why Use It? |
| :--- | :--- |
| **Known Regressions** | If a bug is reported and a ticket is opened, skip the test with a comment (e.g., `// TODO: Fix in JIRA-123`) so the rest of the CI pipeline stays green. |
| **Environment Mismatch** | If a feature (like "Stripe Payments") doesn't work in the `local` environment but works in `staging`, skip it locally. |
| **Platform-Specific Bugs** | When a feature is functional on Chrome but broken on Safari due to a browser engine limitation you can't control. |
| **In-Progress Features** | When you’ve written the test for a feature that hasn't been deployed yet (Test-Driven Development). |



### ⚠️ The "Skip" vs. "Fix" Trap
A common mistake beginners make is using `test.skip()` to hide **flaky tests** (tests that pass sometimes and fail others). 
* **Don't skip flakiness:** If a test is flaky, use `test.fixme()` (which signals the test is broken and needs repair) or investigate the race condition. 
* **Use `.skip()` for intent:** Use it when you *intend* for the test not to run under specific, documented conditions.
