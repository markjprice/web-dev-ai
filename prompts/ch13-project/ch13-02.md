> **Prompt**: Please explain the difference between a Product Requirements Document and a functional specification with examples from a Django web app.

While both documents act as blueprints for your project, they serve different masters: the **PRD** focuses on the "What" and "Why" (for stakeholders and users), while the **Functional Spec** focuses on the "How" (for developers and QA).

Think of the PRD as the **business intent** and the Functional Spec as the **technical contract**.

---

## 1. Product Requirements Document (PRD)
**Focus:** Problems, goals, and user value.
The PRD is usually written by a Product Manager. It defines the features required to solve a user problem without dictating the technical implementation.

### Example: Django "Job Board" App
> **Objective:** Allow users to find work relevant to their skills.
>
> **Requirements:**
> * Users must be able to search for jobs by keyword and location.
> * Employers must be able to post new job listings.
> * The system must notify applicants when their application is viewed.
> * The site must load in under 2 seconds on mobile devices.

---

## 2. Functional Specification (FS)
**Focus:** System behavior, data flow, and logic.
The FS is often a collaborative effort between Product and Engineering. It translates the PRD into specific system behaviors.

### Example: Django "Job Board" App
> **Search Endpoint:**
> * **URL:** `/api/v1/jobs/search/`
> * **Method:** `GET`
> * **Logic:** Query the `JobListing` model. Use `icontains` for the `title` field and `exact` for the `location` field.
> * **Authentication:** Public access allowed.
>
> **Database Schema (Models):**
> ```python
> class JobListing(models.Model):
>     title = models.CharField(max_length=255)
>     description = models.TextField()
>     posted_by = models.ForeignKey(User, on_delete=models.CASCADE)
>     is_active = models.BooleanField(default=True)
> ```
>
> **Task Processing:** > * Use **Celery** with **Redis** to handle email notifications asynchronously when `Application.status` changes to 'Viewed'.

---

## Key Differences at a Glance

| Feature | Product Requirements Document (PRD) | Functional Specification (FS) |
| :--- | :--- | :--- |
| **Primary Audience** | Stakeholders, Designers, Devs | Developers, QA Engineers, DevOps |
| **Key Question** | What are we building and why? | How should the system behave? |
| **Language** | Business-centric / User-centric | Technical / Logical / Systematic |
| **Django Context** | "Users need to upload a profile pic." | "The `Profile` model needs an `ImageField` using S3 storage." |



---

## How They Work Together
In a Django workflow, the PRD might say "We need a subscription system." The Functional Spec will then define the **Stripe API integration**, the specific **Django middleware** needed to check subscription status, and the **URL patterns** for the checkout flow.
