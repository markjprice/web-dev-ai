> **Prompt:** What are the most common mistakes developers make when writing integration tests?

While the Testing Pyramid suggests we should have fewer integration tests than unit tests, the ones we *do* have are often the most difficult to maintain. Because integration tests involve multiple moving parts—databases, APIs, and file systems—they are prone to specific architectural blunders.

Here are the most common mistakes developers make when writing integration tests.

---

## 1. Using "Real" Third-Party Services

One of the most frequent mistakes is hitting actual external APIs (like Stripe, Twilio, or AWS) during a test run.

* **The Problem:** This makes your tests slow, dependent on an internet connection, and potentially expensive. If the third-party service goes down, your builds fail even if your code is perfect.
* **The Fix:** Use tools like **VCR.py** to record real interactions once and replay them, or use local emulators like **LocalStack** (for AWS) or **MailHog** (for email).

---

## 2. Inconsistent Data State (The "Shared DB" Trap)

Integration tests usually share a database. If Test A creates a user named "Alice" and Test B expects the database to be empty, Test B will fail unpredictably.

* **The Problem:** **Non-deterministic tests** (flakiness). Tests pass when run individually but fail when run as a suite.
* **The Fix:** Ensure every test is "atomic." Use database transactions that roll back after every test, or use "cleaner" utilities that truncate tables between runs.

---

## 3. Treating Integration Tests like Unit Tests

Developers often try to test every single edge case (like "what happens if the middle name is an empty string?") at the integration level.

* **The Problem:** Integration tests are slow. If you have 500 integration tests checking basic logic, your CI/CD pipeline will grind to a halt.
* **The Fix:** Use the **"Contract Testing"** mindset. The integration test should verify that the components can *talk* to each other correctly. Leave the exhaustive logic and edge-case checking to your unit tests.

---

## 4. Hardcoding Time and IDs

Tests that rely on `datetime.now()` or hardcoded primary keys (`id=1`) are ticking time bombs.

* **The Problem:** A test might pass today but fail on February 29th (Leap Year) or fail because the database's auto-increment counter didn't reset.
* **The Fix:** Use libraries like **Freezegun** to "pin" the time to a specific moment, and always retrieve IDs dynamically from the objects you created in the "Arrange" phase.

---

## 5. Over-Mocking the "Seams"

Mocking is essential in unit tests, but over-mocking in integration tests defeats the entire purpose of the exercise.

* **The Problem:** If you mock the database and the API client, you aren't actually testing an *integration*; you're just running another unit test with extra overhead.
* **The Fix:** Only mock what you **do not control** (external APIs). Everything you **do control** (your database, your cache, your internal services) should be real to ensure the "plumbing" works.

---

## Summary of Mistakes

| Mistake | Impact | Solution |
| --- | --- | --- |
| **Hitting Live APIs** | Flakiness & Cost | Use VCR or Mocks |
| **Shared State** | Interdependent Failures | Transactions / Cleaners |
| **Logic Overload** | Slow Test Suite | Move logic to Unit Tests |
| **Hardcoded Data** | Environmental Fragility | Dynamic IDs & Time Pinning |
