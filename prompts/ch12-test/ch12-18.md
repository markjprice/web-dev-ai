> **Prompt:** Please show examples of weak AI-generated tests and how to improve them.

Weak AI-generated tests are often "syntactically correct but logically hollow." They focus on achieving high line coverage without actually verifying the integrity of the business logic.

Below are three common patterns of weak AI tests and the strategies used to fix them.

---

## 1. The "Mirror" Test (Confirming a Bug)

This is the most dangerous failure. The AI looks at the code, sees a logical error, and assumes that error is the intended feature.

### Weak Example

```python
# The Source Code (Has a bug: should be 'price * qty')
def calculate_total(price, qty):
    if qty > 10:
        return price * 0.9  # WRONG: Returns price of 1 item, not 10
    return price * qty

# Weak AI Test (Generated by looking at the code)
def test_calculate_total_discount():
    # The AI sees the bug and writes the assertion to match it
    result = calculate_total(100, 11)
    assert result == 90.0  # Pass-through failure!

```

### The Fix: Requirement-Based Oracles

Force the AI to calculate the expected value independently of the function's return.

**Improved Test:**

```python
@pytest.mark.parametrize("price, qty, expected", [
    (100, 5, 500),   # Standard: 100 * 5
    (100, 20, 1800), # Discounted: (100 * 20) * 0.9 = 1800
])
def test_calculate_total_fixed(price, qty, expected):
    assert calculate_total(price, qty) == expected

```

---

## 2. The "Happy Path" Tunnel Vision

Weak AI tends to generate the most obvious inputs (, "John Doe") while ignoring the "dark corners" of the application logic.

### Weak Example

```python
# Weak AI Test for a username validator
def test_username_is_valid():
    assert validate_username("john_doe") is True
    assert validate_username("admin123") is True

```

### The Fix: Adversarial Boundary Analysis

Provide the AI with a list of "destructive" categories to test against.

**Improved Test:**

```python
@pytest.mark.parametrize("username, expected", [
    ("a", False),              # Too short
    ("very_long_name" * 10, False), # Too long
    ("", False),               # Empty
    ("user!name", False),      # Special characters
    (None, False),             # Null input
    ("  trimmed  ", True),     # Whitespace handling
])
def test_username_boundaries(username, expected):
    assert validate_username(username) == expected

```

---

## 3. The "Fragile Mock" (Deep Coupling)

AI often mocks internal implementation details (private methods) rather than public interfaces. If you rename a private function, the test breaks even if the feature still works.

### Weak Example

```python
# Weak AI Test using excessive internal mocking
def test_send_email_logic(mocker):
    # The AI mocks the internal private method '_format_string'
    mock_format = mocker.patch('myapp.services.EmailService._format_string')
    mock_format.return_value = "Hello"
    
    service = EmailService()
    service.send("test@ex.com")
    
    # This test fails if we ever refactor how strings are formatted
    mock_format.assert_called_once()

```

### The Fix: Behavior-Driven Mocking

Only mock at the "I/O boundaries" (Database, Network, File System).

**Improved Test:**

```python
def test_send_email_behavior(mocker):
    # Mock only the external outgoing SMTP call
    mock_smtp = mocker.patch('myapp.services.smtplib.SMTP')
    
    service = EmailService()
    service.send("test@ex.com")
    
    # We only care that the external side effect happened correctly
    assert mock_smtp.return_value.sendmail.called

```

---

## Summary of Improvements

| Weakness | Symptom | Improvement Strategy |
| --- | --- | --- |
| **Pass-through** | Test passes on buggy code. | Use **Requirement-First** oracles. |
| **Surface Level** | Misses edge cases. | Use **Parametrized** boundary testing. |
| **Brittle** | Breaks during refactoring. | Mock **Side Effects**, not internal logic. |
