> **Prompt:** What makes a good unit test beyond Arrange–Act–Assert? Give examples of bad but syntactically correct tests.

While the **Arrange-Act-Assert (AAA)** pattern provides a clean structural template, a "good" unit test is defined by its ability to act as a safety net without becoming a maintenance burden.

Beyond the structure, a high-quality test must possess specific architectural traits.

---

## 1. The "FIRST" Principles of Unit Testing

To move beyond basic syntax, elite teams typically follow the **FIRST** acronym:

* **Fast:** Tests should run in milliseconds. If they take seconds, developers will stop running them locally, and the "feedback loop" breaks.
* **Independent:** Tests must not depend on each other. You should be able to run them in any order or in parallel.
* **Repeatable:** A test should yield the same result every time, regardless of the environment (no reliance on the current date, random numbers, or external APIs).
* **Self-Validating:** The test must have a clear boolean output (Pass/Fail). No manual interpretation of logs required.
* **Thorough/Timely:** Cover edge cases (nulls, empty strings, max values), not just the "happy path."

---

## 2. Key Qualitative Traits

A great test also serves as **living documentation**. If a new developer joins the team, they should be able to read your tests to understand the business requirements.

### Low Implementation Knowledge

A test should verify *what* the code does, not *how* it does it. If you refactor a function's internal logic but the output stays the same, the test should still pass. If the test breaks during a refactor, it is "overspecified" and tightly coupled to the implementation.

### Single Responsibility

Each test should ideally check one specific behavior or logical branch. If a test has ten different assertions covering five different scenarios, a failure won't tell you exactly what went wrong.

## 3. "Bad" but Syntactically Correct Examples

These examples will compile and run, but they provide a false sense of security or create "technical debt" in your test suite.

### Example A: The "Evergreen" (No-Assertion) Test

This test passes as long as the code doesn't crash, but it doesn't actually verify the result.

```javascript
// BAD: Syntactically correct, but proves nothing.
test('calculateTotal should work', () => {
  const cart = new ShoppingCart();
  cart.addItem({ price: 10, qty: 2 });
  cart.calculateTotal(); 
  // Missing: expect(cart.total).toBe(20);
});

```

### Example B: The "Brittle" Implementation Test

This test focuses on *how* the object works internally rather than the result. If you change a private variable name, the test breaks.

```javascript
// BAD: Tightly coupled to internal state.
test('adds user to internal array', () => {
  const service = new UserService();
  service.addUser("Alice");
  
  // This is bad because it inspects a private/internal property
  expect(service._internalUserList.length).toBe(1); 
});

```

### Example C: The "Logic-Heavy" Test

When you put `if` statements or `loops` inside a test, you are essentially writing code that needs its own tests.

```javascript
// BAD: Complex logic inside the test itself.
test('processes all items', () => {
  const items = [1, 2, 3];
  const results = processItems(items);
  
  for (let i = 0; i < results.length; i++) {
    if (results[i] > 0) {
      expect(results[i]).toBe(items[i] * 2);
    }
  }
});

```

---

## 4. The "Test Double" Strategy

A common mistake in bad unit tests is hitting a real database or a real network. This violates the **Repeatable** and **Fast** rules. Good tests use "Mocks" or "Stubs" to simulate these dependencies.

* **Stub:** Provides canned answers to calls made during the test.
* **Mock:** Used to verify interactions (e.g., "Did the email service get called exactly once?").
