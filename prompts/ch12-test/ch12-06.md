> **Prompt:** Why do many professional Django teams prefer `pytest` over `unittest`? Please include a comparison of the same Django test written in `unittest` style and `pytest` style.

While Django's built-in testing framework (based on `unittest`) is robust, many professional teams migrate to **pytest** (specifically using the `pytest-django` plugin) because it reduces "boilerplate" code and offers more powerful debugging tools.

---

## 1. Why Teams Switch to Pytest

* **Boilerplate Reduction:** In `unittest`, you must wrap everything in a class and use specific methods like `self.assertEqual`. Pytest uses plain Python `assert` statements.
* **Fixtures vs. `setUp`:** `unittest` uses `setUp()` and `tearDown()`, which run for every test in a class. Pytest uses **Fixtures**, which are modular, reusable, and can have different scopes (e.g., once per session, once per module).
* **Powerful CLI:** Pytest makes it much easier to run specific tests (`-k "logic_name"`), stop on the first failure (`-x`), or drop into a debugger on failure (`--pdb`).
* **Advanced Features:** Pytest supports **Parameterization** (running the same test with different inputs) and has a massive ecosystem of plugins for things like code coverage, profiling, and mocking.

---

## 2. Code Comparison: The Same Test

Imagine we are testing a simple Django model method that calculates a discounted price.

### The `unittest` Style (Django Default)

Notice the mandatory class structure and the specific assertion methods.

```python
from django.test import TestCase
from .models import Product

class ProductTest(TestCase):
    def setUp(self):
        self.product = Product.objects.create(name="Laptop", price=1000)

    def test_discounted_price(self):
        discounted = self.product.get_discounted_price(0.2) # 20% off
        self.assertEqual(discounted, 800)

```

### The `pytest` Style

Notice the lack of classes (optional), the use of a reusable fixture, and a standard Python `assert`.

```python
import pytest
from .models import Product

@pytest.fixture
def product(db):
    return Product.objects.create(name="Laptop", price=1000)

def test_discounted_price(product):
    assert product.get_discounted_price(0.2) == 800

```

---

## 3. Key Differences in the Code

| Feature | `unittest` (Django) | `pytest` |
| --- | --- | --- |
| **Structure** | Must use Classes inheriting from `TestCase`. | Can use simple functions. |
| **Database** | Handled automatically by `TestCase`. | Requires `@pytest.mark.django_db` or a `db` fixture. |
| **Data Setup** | `setUp()` method (runs every time). | `fixtures` (reusable across files). |
| **Assertions** | `self.assertEqual(a, b)` | `assert a == b` |
| **Readability** | High overhead/nesting. | Clean and "Pythonic." |

---

## 4. The "Hidden" Killer Feature: Parameterization

If you wanted to test five different discount rates in `unittest`, you'd likely write five tests or a loop. In `pytest`, it looks like this:

```python
@pytest.mark.parametrize("rate, expected", [
    (0.1, 900),
    (0.5, 500),
    (1.0, 0),
])
def test_various_discounts(product, rate, expected):
    assert product.get_discounted_price(rate) == expected

```

Pytest will treat this as **three separate tests** in your reports, making it much easier to see exactly which case failed.
