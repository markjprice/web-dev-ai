> **Prompt:** What is parametrized testing in `pytest` and when is it useful?

### What is Parametrized Testing?

In **pytest**, parametrization allows you to run the same test function multiple times using different sets of input data. Instead of writing separate test functions for every possible scenario, you define a single "template" and provide a list of arguments that pytest will inject into the test one by one.

Each set of arguments is treated as a **unique test case**. If one set fails, it doesn't stop the others from running, and the test report will show exactly which input caused the failure.

---

### How to Implement It

You use the `@pytest.mark.parametrize` decorator. The first argument is a string containing the names of the parameters, and the second is a list (or tuple) containing the data.

```python
import pytest

# The function we want to test
def is_even(number):
    return number % 2 == 0

# Parametrized test
@pytest.mark.parametrize("input_val, expected_result", [
    (2, True),
    (7, False),
    (0, True),
    (-4, True),
    (11, False),
])
def test_is_even(input_val, expected_result):
    assert is_even(input_val) == expected_result

```

---

### When is it Useful?

#### 1. Testing Boundary Conditions (Edge Cases)

If you have a function that handles numeric ranges or string lengths, you can test the "happy path," the minimum value, the maximum value, and invalid inputs (like `null` or `0`) all in one block.

#### 2. Cross-Browser or Cross-Platform Testing

In web development, you can parametrize a browser fixture to run the same UI test suite against Chrome, Firefox, and Safari without duplicating your test logic.

#### 3. Mathematical or Logical Validations

As seen in the `is_even` example, any logic that depends on a "truth table" or a set of known inputs and outputs is a perfect candidate. It ensures that fixing a bug for one specific input doesn't accidentally break another.

#### 4. Reducing "Copy-Paste" Maintenance

If you find yourself writing `test_login_admin`, `test_login_editor`, and `test_login_guest` where the only difference is the user role, you should parametrize them. This ensures that if the login flow changes, you only have to update **one** test function instead of three.

---

### Comparison: Loops vs. Parametrization

It is a common mistake to use a `for` loop inside a standard test. Here is why parametrization is superior:

| Feature | Using a `for` loop in a test | Using `@pytest.mark.parametrize` |
| --- | --- | --- |
| **Failure Isolation** | If the first item fails, the loop stops. Remaining items aren't tested. | If one case fails, pytest continues to run the rest. |
| **Reporting** | Counts as **1 test**. | Counts as **N tests** (one for each data set). |
| **Readability** | Logic and data are mixed together. | Data is cleanly separated in the decorator. |
| **Debugging** | Hard to tell which specific input failed without logs. | The CLI output shows the specific parameters for the failure. |

---

### Advanced Use: Stacked Parametrization

You can even "stack" decorators to create a **Cartesian Product** of tests.

```python
@pytest.mark.parametrize("user", ["admin", "guest"])
@pytest.mark.parametrize("page", ["/dashboard", "/settings"])
def test_access(user, page):
    # This will run 4 times total:
    # admin + /dashboard, admin + /settings, guest + /dashboard, guest + /settings
    ...

```
