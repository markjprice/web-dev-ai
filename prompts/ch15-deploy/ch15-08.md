> **Prompt:** What does Django’s `check --deploy` command actually validate internally?

The `python manage.py check --deploy` command is your final automated "security guard." While standard system checks look for code errors (like broken model relationships), the `--deploy` flag activates a specific suite of **security-focused checks** that only matter in a live environment.

Internally, it uses Django’s **System Check Framework** to validate your `settings.py` against known production risks. Here is exactly what it is looking for:

---

## 1. The "Big Three" Critical Fixes
These are the most common ways Django apps get hacked. The command checks:
* **`DEBUG = False`:** It ensures you aren't leaking your source code, local variables, and database schema via those helpful "Yellow Screen of Death" error pages.
* **`SECRET_KEY`:** It validates that your key is long, complex, and (crucially) not the default one provided by `startproject`.
* **`ALLOWED_HOSTS`:** It checks that this isn't empty or a wildcard (`*`). This prevents **HTTP Host Header attacks**, where an attacker tricks your app into generating links to a malicious domain.

## 2. HTTPS & Secure Transmission
If you’re in production, your data should be encrypted. The command validates:
* **`SECURE_SSL_REDIRECT`:** Ensures all HTTP traffic is bumped to HTTPS.
* **`SECURE_HSTS_SECONDS`:** Checks if you've enabled **HTTP Strict Transport Security**, which tells browsers "Only ever talk to this site via HTTPS."
* **`SESSION_COOKIE_SECURE` & `CSRF_COOKIE_SECURE`:** These ensure your sensitive login and CSRF cookies are *only* sent over encrypted (HTTPS) connections.

## 3. Browser-Side Protection (Security Headers)
It checks if you've told the user's browser to protect itself:
* **`SECURE_CONTENT_TYPE_NOSNIFF`:** Sets the `X-Content-Type-Options: nosniff` header. This prevents the browser from trying to "guess" a file's type, which stops attackers from disguising a script as a harmless image.
* **`X_FRAME_OPTIONS`:** Usually set to `DENY` or `SAMEORIGIN`. This prevents **Clickjacking**, where your site is loaded in an invisible iframe on another site to steal clicks.



---

## What it DOES NOT validate
It is important to know the limits of this command. It is a **static check of settings**, not a penetration test. It won't find:
* **Weak Passwords:** It doesn't check if your admin password is "password123."
* **Server-Level Misconfigs:** It can't see if your Nginx config is broken or if your database port is accidentally open to the public internet.
* **Vulnerable Dependencies:** It doesn't scan your `requirements.txt` for outdated packages (you need tools like `safety` or `pip-audit` for that).

---

## How to use it effectively
You should run this as part of your **CI/CD pipeline** (the automated process that deploys your code).

```bash
# In your deployment script
python manage.py check --deploy --settings=myproject.settings.production
```

If it returns **SystemCheckError**, your deployment should fail automatically. This prevents you from accidentally "going live" with a dangerous configuration.
