> **Prompt:** Please walk me through migrating a Django project from SQLite to PostgreSQL safely.

Migrating from SQLite to PostgreSQL is a rite of passage for Django developers. Since PostgreSQL is a separate service, this isn't as simple as renaming a file; you are moving data between two different database engines with different ways of handling types and constraints.

Here is the safest workflow to ensure you don't lose data.

---

## Phase 1: Preparation
Before touching your data, you need to prepare your environment for PostgreSQL.

1.  **Install PostgreSQL:** Ensure Postgres is running on your machine or server.
2.  **Create the Database:** Use the Postgres CLI or a tool like pgAdmin to create a fresh database.
    ```sql
    CREATE DATABASE my_project_db;
    CREATE USER my_user WITH PASSWORD 'my_password';
    ALTER ROLE my_user SET client_encoding TO 'utf8';
    ALTER ROLE my_user SET default_transaction_isolation TO 'read committed';
    ALTER ROLE my_user SET timezone TO 'UTC';
    GRANT ALL PRIVILEGES ON DATABASE my_project_db TO my_user;
    ```
3.  **Install the Driver:** Django needs a bridge to talk to Postgres.
    ```bash
    pip install psycopg2-binary
    ```

---

## Phase 2: Extracting Data (The "Dump")
We will use Django’s `dumpdata` command. This exports your SQLite data into a generic JSON format that Postgres can understand.

1.  **Clean up content types:** SQLite often has "stale" records for system-generated data. Run this to avoid primary key conflicts later:
    ```bash
    python manage.py shell
    # Inside the shell:
    from django.contrib.contenttypes.models import ContentType
    ContentType.objects.all().delete()
    quit()
    ```
2.  **Dump the data:**
    ```bash
    python manage.py dumpdata --exclude auth.permission --exclude contenttypes > data.json
    ```
    *> **Note:** We exclude permissions and contenttypes because Django will recreate these automatically when we migrate to the new DB.*

---

## Phase 3: Switching the Engine
Now, tell Django to look at PostgreSQL instead of the SQLite file.

1.  **Update `settings.py`:**
    ```python
    DATABASES = {
        'default': {
            'ENGINE': 'django.db.backends.postgresql',
            'NAME': 'my_project_db',
            'USER': 'my_user',
            'PASSWORD': 'my_password',
            'HOST': 'localhost',
            'PORT': '5432',
        }
    }
    ```

---

## Phase 4: Loading Data (The "Restore")
This is where the magic (and potential errors) happens.

1.  **Create the schema:** Run migrations on the empty Postgres database so the tables exist.
    ```bash
    python manage.py migrate
    ```
2.  **Load the data:**
    ```bash
    python manage.py loaddata data.json
    ```



---

## Phase 5: Common Pitfalls & Fixes

### 1. Sequential ID Desync (Postgres Specific)
Unlike SQLite, Postgres uses "Sequences" to keep track of the next ID number. Sometimes `loaddata` doesn't update these, and you'll get an error when you try to create a new record: `duplicate key value violates unique constraint`.

**The Fix:** Run this command to reset the ID counters:
```bash
python manage.py sqlsequencereset my_app_name | python manage.py dbshell
```

### 2. Case Sensitivity
SQLite is often case-insensitive for unique constraints; PostgreSQL is strictly case-sensitive. If you have two users named "Alex" and "alex", SQLite might have allowed it, but Postgres will bark during the `loaddata` phase. You’ll need to clean your data in the JSON file if this happens.

### 3. Boolean and Date differences
If you have custom SQL or specific fields, Postgres is much stricter about types. Ensure your `data.json` uses standard ISO formats for dates.

---

## Summary Checklist
* [ ] Backup your `db.sqlite3` file (Copy it to a safe place!).
* [ ] Dump data to JSON (excluding system tables).
* [ ] Update `settings.py` to Postgres.
* [ ] `migrate` to create empty tables.
* [ ] `loaddata` to fill the tables.
* [ ] Reset sequences using `sqlsequencereset`.
