> **Prompt:** Please write a bash script to deploy a Django app safely with zero downtime.

To achieve a true **zero-downtime deployment** on a single server, we use Gunicorn's ability to handle a `HUP` (hang-up) signal. This tells the master process to spawn new workers with the updated code *before* gracefully shutting down the old ones.

### The Zero-Downtime Strategy
1.  **Preparation:** Pull code and install dependencies in the background.
2.  **Migration:** Run database migrations (Postgres/SQLite).
3.  **Static Files:** Collect CSS/JS so Nginx can find them.
4.  **The "HUP" Reload:** Signal Gunicorn to swap workers without dropping a single socket connection.



---

### The Bash Script (`deploy.sh`)

Save this script in your project root on the server (e.g., `/var/www/my-app/deploy.sh`).

```bash
#!/bin/bash

# --- CONFIGURATION ---
APP_NAME="my_django_project"
PROJECT_DIR="/var/www/my-app"
VENV_DIR="$PROJECT_DIR/venv"
PID_FILE="$PROJECT_DIR/gunicorn.pid"
LOG_FILE="$PROJECT_DIR/logs/deploy.log"

# Function to log messages
log_message() {
    echo "$(date +'%Y-%m-%d %H:%M:%S') - $1" | tee -a "$LOG_FILE"
}

# Exit immediately if a command fails
set -e

log_message "🚀 Starting Zero-Downtime Deployment for $APP_NAME..."

cd "$PROJECT_DIR"

# 1. Pull latest code from GitHub/GitLab
log_message "📥 Pulling latest code from main..."
git pull origin main

# 2. Activate Virtual Environment
source "$VENV_DIR/bin/activate"

# 3. Install/Update Requirements
log_message "📦 Updating dependencies..."
pip install -r requirements.txt --quiet

# 4. Run Database Migrations
log_message "🗄️ Running database migrations..."
python manage.py migrate --noinput

# 5. Collect Static Files
log_message "🎨 Collecting static files..."
python manage.py collectstatic --noinput --clear

# 6. Zero-Downtime Reload
if [ -f "$PID_FILE" ]; then
    log_message "♻️  Sending HUP signal to Gunicorn (Master PID: $(cat $PID_FILE))..."
    # SIGHUP tells Gunicorn to spawn new workers and then phase out old ones
    kill -HUP $(cat "$PID_FILE")
    log_message "✅ Gunicorn reload initiated."
else
    log_message "⚠️  Gunicorn PID file not found. Starting fresh..."
    # Ensure you use the --pid flag when starting Gunicorn!
    gunicorn --pid "$PID_FILE" \
             --bind 127.0.0.1:8000 \
             --workers 3 \
             --daemon \
             "$APP_NAME.wsgi:application"
    log_message "✅ Gunicorn started from scratch."
fi

log_message "🎉 Deployment Complete!"
```

---

### Critical Requirements for this to Work

To make this script functional, your **Gunicorn systemd service** (or your initial start command) **must** use a PID file. If Gunicorn doesn't save its process ID to a file, the script won't know who to signal.

#### Example Gunicorn Service Configuration (`/etc/systemd/system/gunicorn.service`):
```ini
[Service]
User=ubuntu
Group=www-data
WorkingDirectory=/var/www/my-app
# The %p is the PID of the main process
ExecStart=/var/www/my-app/venv/bin/gunicorn \
    --pid /var/www/my-app/gunicorn.pid \
    --access-logfile - \
    --workers 3 \
    --bind unix:/run/gunicorn.sock \
    my_django_project.wsgi:application
```

### Why this is "Safe"
* **The HUP Signal:** Unlike `systemctl restart gunicorn` (which stops the service and then starts it, causing a 1–2 second "502 Bad Gateway"), `kill -HUP` keeps the listening socket open. The user's request simply waits a few milliseconds longer while a new worker picks it up.
* **`set -e`:** If the `pip install` fails or `migrate` crashes, the script stops immediately and won't attempt to reload the server with broken code.

### A Note on Database Migrations
While the **code** reload is zero-downtime, **database migrations** are logically tricky. If you delete a database column that the *old* workers still expect to see, you will get errors during the 5-second window while the workers are swapping.
* **Best Practice:** Always perform "Additive" migrations (Add column, Deploy, then Delete old column in a separate deploy later).
