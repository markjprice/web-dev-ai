> **Prompt:** How can I automate backups and test restoring a SQLite database?

Automating backups for a SQLite database is surprisingly powerful because the database is "just a file." In 2026, you have two primary ways to do this: the **Django-native way** (using a battle-tested package) or the **Modern-SRE way** (using continuous replication).

---

## 1. The Django-Native Way: `django-dbbackup`
This is the most "beginner-friendly" method. It creates a snapshot of your database and sends it to a secure location like Amazon S3, Dropbox, or a local folder.

### Step 1: Install and Configure
```bash
pip install django-dbbackup django-storages[boto3]
```

In your `settings.py`:
```python
INSTALLED_APPS = [
    ...
    'dbbackup',
]

# Example: Backing up to a local folder (or use S3 for production)
DBBACKUP_STORAGE = 'django.core.files.storage.FileSystemStorage'
DBBACKUP_STORAGE_OPTIONS = {'location': '/home/ubuntu/backups/'}
```

### Step 2: Automate with Crontab
To run a backup every night at 2:00 AM, add this to your server's crontab (`crontab -e`):
```bash
0 2 * * * /path/to/venv/bin/python /path/to/project/manage.py dbbackup --clean
```
*The `--clean` flag automatically deletes old backups based on your retention settings.*

---

## 2. The Modern Way: Litestream (Continuous Backup)
If you can't afford to lose even 10 minutes of data, use **Litestream**. It is a separate tool that "watches" your SQLite file and streams every single change to S3 in real-time.

* **How it works:** It acts as a sidecar process. If your server dies, you just run `litestream restore` and it pulls the latest version from the cloud.
* **Why use it:** It provides **Point-in-Time Recovery**, meaning you can restore your database to exactly how it looked at 10:14 AM yesterday.



---

## 3. How to Test Your Restoration
A backup is worthless if it doesn't work. You should automate the testing of your restore process using a simple script.

### The "Drill" Script
Create a script called `test_restore.sh`. It should:
1.  Create a temporary "test" database.
2.  Restore the latest backup into it.
3.  Run a simple Django command to verify data exists.

```bash
#!/bin/bash
set -e

echo "🧪 Testing latest backup..."

# 1. Restore the backup to a temporary file
python manage.py dbrestore --noinput --database=default --filepath=latest_backup.psql

# 2. Verify the database isn't empty
USER_COUNT=$(python manage.py shell -c "from django.contrib.auth.models import User; print(User.objects.count())")

if [ "$USER_COUNT" -gt 0 ]; then
    echo "✅ Success: Found $USER_COUNT users in the restored backup."
else
    echo "❌ Failure: Restored database is empty!"
    exit 1
fi
```

### Why test restores automatically?
* **Corruption:** Sometimes the backup process itself fails, leaving you with a 0-byte file.
* **Schema Mismatch:** If you migrated your models but are trying to restore an old backup, Django might crash. Testing ensures the data matches your current code.
