> **Prompt**: Please show me how to write accessible HTML for images, links, and forms.

Writing accessible HTML isn't just about using the right tags; it's about providing the necessary context for users who might be navigating your site using a screen reader, a keyboard, or even a slow internet connection.

Here is how to handle the three most common elements:

---

## 1. Images: The `alt` Attribute
The `alt` (alternative text) attribute is the most important part of an image. If the image fails to load or if a user is blind, this text describes what is happening.

### Good Practice
* **Informative Images:** Describe the content.
* **Functional Images:** Describe the action (e.g., "Search" for a magnifying glass icon).
* **Decorative Images:** Use an empty `alt=""`. This tells screen readers to skip it entirely.

```html
<img src="golden-retriever.jpg" alt="A golden retriever puppy sitting in green grass.">

<img src="divider-line.png" alt="">

<a href="/search">
  <img src="glass.png" alt="Search the site">
</a>
```


---

## 2. Links: Descriptive Text
The golden rule for links is **never use "Click Here" or "Read More."** Screen reader users often pull up a "Links List" to navigate quickly. If your list says "Click Here, Click Here, Click Here," the user has no idea where those links go.

### Good vs. Bad Examples

| Accessibility | Code | Why? |
| :--- | :--- | :--- |
| **Bad** | `To see our prices, <a href="/pricing">click here</a>.` | "Click here" provides no context in a links list. |
| **Good** | `View our <a href="/pricing">full pricing guide</a>.` | The link text itself describes the destination. |
| **Better** | `<a href="/report.pdf" download>Download the 2026 Annual Report (PDF, 2MB)</a>` | Tells the user the file type and size before they click. |

---

## 3. Forms: Labels and Groups
Forms are where accessibility often breaks. A user needs to know exactly what information is required for every input.

### Key Rules
* **Always use `<label>`:** Never rely on `placeholder` text alone. Placeholders disappear when the user starts typing, which is difficult for users with cognitive disabilities.
* **The `for` attribute:** The `for` attribute of the label must match the `id` of the input. This makes the label clickable, which helps users with motor impairments.
* **Group related items:** Use `<fieldset>` and `<legend>` for groups of checkboxes or radio buttons.

### Accessible Form Example
```html
<form>
  <div>
    <label for="user-email">Email Address:</label>
    <input type="email" id="user-email" name="email" required>
  </div>

  <fieldset>
    <legend>Choose your favorite fruit:</legend>
    
    <input type="radio" id="apple" name="fruit" value="apple">
    <label for="apple">Apple</label>

    <input type="radio" id="banana" name="fruit" value="banana">
    <label for="banana">Banana</label>
  </fieldset>

  <button type="submit">Submit Preferences</button>
</form>
```


---

## Why This Matters
1. **Focus Management:** Using proper labels and inputs allows users to "Tab" through your form in a logical order.
2. **Error Handling:** When you use the `required` attribute or `aria-describedby`, screen readers can alert the user exactly where they made a mistake.
3. **Target Size:** Labels increase the clickable area of an input, making it easier for people with tremors or those on mobile devices to select small checkboxes.

# How to add aria-labels for more complex components

Icon-only buttons are a classic accessibility trap. Since there is no visible text, a screen reader will simply announce "Button," leaving the user to guess whether it deletes their account or sends a message.

To fix this, we use **ARIA (Accessible Rich Internet Applications)**.

---

## 1. Using `aria-label`
The `aria-label` attribute provides a text string that labels the element. This label is invisible to sighted users but is read aloud by assistive technology.

### The "Broken" Way
```html
<button onclick="deleteItem()">
  <i class="fa-trash"></i> </button>
```
* **Result:** Screen reader says: *"Button."*

### The Accessible Way
```html
<button onclick="deleteItem()" aria-label="Delete this item">
  <i class="fa-trash" aria-hidden="true"></i>
</button>
```
* **Result:** Screen reader says: *"Delete this item, button."*
* **Note:** We use `aria-hidden="true"` on the icon itself so the screen reader doesn't try to "read" the decorative font-icon code.



---

## 2. Using `aria-labelledby`
If the text that describes the button is already visible somewhere else on the screen (like a section title), you can link them using an ID.

```html
<h2 id="settings-title">Account Settings</h2>

<button aria-labelledby="settings-title">
  <img src="gear-icon.svg" alt="">
</button>
```

---

## 3. The "Visually Hidden" Technique
Sometimes, developers prefer to put actual text inside the button but hide it from sighted users using CSS. This is often considered the "gold standard" because it's the most robust across different browsers.

**The HTML:**
```html
<button>
  <i class="fa-envelope" aria-hidden="true"></i>
  <span class="sr-only">Send Message</span>
</button>
```

**The CSS (standard "screen-reader only" utility):**
```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  border: 0;
}
```

---

## Summary of ARIA for Components

| Attribute | Purpose | Best Used For... |
| :--- | :--- | :--- |
| **`aria-label`** | Direct text label | Close buttons (X), Hamburger menus, Icon-only social links. |
| **`aria-labelledby`** | References an ID | Buttons that relate to a specific heading or label nearby. |
| **`aria-describedby`** | Provides extra info | "Submit" buttons that need to mention "Terms and Conditions apply." |
| **`aria-hidden`** | Hides from screen readers | Icons, decorative shapes, or images that are purely visual. |
