> **Prompt**: Please explain the difference between semantic HTML and non-semantic HTML with real-world examples and why it matters for accessibility.

Think of semantic HTML like a well-labeled filing cabinet. Without labels, you can still put papers in the drawers, but finding what you need becomes a nightmare for anyone else—especially for those who can't see the cabinet.

In web development, **semantics** refers to the meaning of the code rather than just its appearance.

---

## Semantic vs. Non-Semantic HTML

### 1. Non-Semantic HTML
These elements tell the browser nothing about their content. They are essentially empty "boxes" used for styling and layout.
* **Examples:** `<div>` and `<span>`.
* **The Issue:** A screen reader sees a `<div>` and simply says "group" or "extra content." It doesn't know if that box is a header, a navigation menu, or a footer.

### 2. Semantic HTML
These elements clearly describe their purpose to both the browser and the developer.
* **Examples:** `<header>`, `<nav>`, `<main>`, `<footer>`, `<article>`, and `<button>`.
* **The Benefit:** They provide a roadmap for assistive technologies, allowing users to jump directly to the "main" content or find the "navigation" quickly.



---

## Real-World Comparison

Imagine you are building a "Submit" button.

### The Non-Semantic Way (Bad Practice)
```html
<div class="my-button" onclick="submitForm()">
  Submit
</div>
```
* **The Problem:** To a sighted user, CSS might make this look like a button. However, a user navigating via keyboard cannot "Tab" to this element because a `div` isn't focusable by default. A screen reader won't announce it as a button, so a blind user won't know it's interactive.

### The Semantic Way (Best Practice)
```html
<button type="submit">
  Submit
</button>
```
* **The Solution:** The `<button>` tag automatically handles keyboard focus (the "Enter" and "Space" keys work by default) and tells screen readers exactly what it is. No extra JavaScript or "hacks" are required to make it accessible.

---

## Why It Matters for Accessibility

Using semantic HTML is the single most effective way to make a website accessible. It creates an **Accessibility Tree**, which is a version of the page designed specifically for screen readers.

| Feature | Non-Semantic (`div`/`span`) | Semantic (`header`/`nav`/`section`) |
| :--- | :--- | :--- |
| **Keyboard Navigation** | Broken (requires manual fixes) | Works out of the box |
| **Screen Readers** | See a "wall of text" | Can skip to specific landmarks |
| **SEO** | Harder for bots to index | Search engines prioritize semantic tags |
| **Code Clarity** | "Div-soup" (hard to read) | Clean, meaningful structure |

---

## A Common Landmark Structure
A properly structured semantic page looks like this:

```html
<header>
  <nav> </nav>
</header>

<main>
  <section>
    <h1>Main Topic</h1>
    <article> Specific blog post or news story </article>
  </section>
  
  <aside> Sidebar with related info </aside>
</main>

<footer> Copyright and contact info </footer>
```
