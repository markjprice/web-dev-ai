> **Prompt**: How do I check if my color contrast meets WCAG standards? What CSS techniques improve accessibility beyond color contrast?

Checking for color contrast is a critical step in making your web projects accessible to everyone, including users with low vision or color blindness. **WCAG 2.1** standards generally require a contrast ratio of at least **4.5:1** for normal text and **3:1** for large text.

---

## 1. How to Check Color Contrast

### Using Browser DevTools (The Quickest Way)
Both Chrome and Firefox have built-in inspectors that calculate contrast ratios instantly:
1.  Right-click an element and select **Inspect**.
2.  In the **Styles** pane, click the small color square (swatch) next to the `color` property.
3.  The color picker will show the **Contrast Ratio** section. It will display a green checkmark if it passes AA or AAA standards.



### Online Tools
If you are in the design phase and haven't written code yet, use these:
* **WebAIM Contrast Checker:** The industry standard. You input hex codes, and it gives you a Pass/Fail.
* **Adobe Color:** Includes a "Contrast Checker" tab specifically for accessibility.

## 2. CSS Techniques to Improve Accessibility

Accessibility (A11y) goes far beyond just colors. Here are four techniques to make your CSS more inclusive:

### A. Don’t Rely on Color Alone
Never use color as the *only* way to convey information (e.g., making an error message red). Users with color blindness might not distinguish it from other text.
* **Technique:** Use icons, text labels, or patterns.
* **Example:** For links within a paragraph, ensure they are **underlined** so they are distinguishable by shape, not just color.

### B. Focus Indicators (`:focus-visible`)
Users who navigate via keyboard (using the Tab key) need to know where they are on the page. Removing the default "blue ring" without replacing it is a major accessibility failure.
* **Technique:** Use `:focus-visible` to style the focus state only when a user is using a keyboard.
```css
button:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
}
```

### C. Responsive Text with Relative Units
Hard-coding font sizes in `px` can prevent users from scaling text using their browser settings.
* **Technique:** Use `rem` for font sizes and `em` for spacing. This ensures that if a user increases their default browser font size to 200%, your layout scales gracefully.
```css
body {
  font-size: 1rem; /* Usually defaults to 16px */
}
```

### D. Respect User Preferences (`prefers-reduced-motion`)
Some users experience motion sickness or seizures from parallax scrolling and heavy animations.
* **Technique:** Use a media query to disable or simplify animations for those who have requested reduced motion in their OS settings.
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

## 3. The "Visually Hidden" Utility
Sometimes you need to provide text specifically for Screen Readers (like a label for a search icon) that shouldn't appear on the screen. **Do not use `display: none`**, as screen readers will ignore it too.

**The "SR-Only" Pattern:**
```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```
